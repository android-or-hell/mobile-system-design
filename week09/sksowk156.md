* API 경계를 먼저 정하고 내부 구현 최적화하기

## 4.1 UI와 다른 구현 중 어디에 집중할 것인가

초기 프로토타입이 준비되면 두 가지 방향 중 하나를 선택할 수 있다.

1. **더 깊은 계층으로 내려가 `CourseService`와 백엔드 통합을 진행한다.**
2. **위쪽 계층으로 올라가 `Course`를 실제 UI와 연결한다.**

### 4.1.1 UI에 집중하기

UI 우선 접근은 빠르게 눈에 보이는 결과물을 만들 수 있다는 장점이 있다.

* 사용자 테스트
* UI 반복 개선
* 내비게이션 흐름 검토
* 실제 빌드 배포

하지만 UI부터 완성하면 앱이 실제보다 일찍 완성된 것처럼 보일 수 있다. 이후 데이터 처리, 파싱, 백엔드 연동처럼 보이지 않는 작업이 남으면 개발 속도가 갑자기 느려진 것처럼 느껴질 수 있다.

따라서 **완성도 높은 UI는 미루고, 최소한의 UI만 실제 데이터 흐름에 연결하는 방식**을 고려할 수 있다.

또한 UI를 모두 만든 뒤 통합을 시작하면 모바일과 백엔드 사이의 불일치를 늦게 발견할 위험이 있다.

### 4.1.2 더 깊은 계층으로 내려가기

통합 우선 접근은 백엔드와의 통신 방식과 요구사항을 일찍 검증하는 방법이다.

이 과정에서 다음 문제를 발견할 수 있다.

* 서로 맞지 않는 오류 코드
* 오래되었거나 없는 백엔드 문서
* 최신 상태가 아닌 스테이징 환경
* 디자인과 다른 필수·선택 필드
* 누락된 API
* 외부 서비스의 인증이나 비용 문제

**통합 문제는 늦게 발견할수록 수정 비용이 커지므로 가능한 한 일찍 확인하는 것이 좋다.**

### 4.1.3 백엔드 통합을 우선하라

백엔드 통합을 일찍 시작하면 모바일과 백엔드 개발자가 잘못된 가정을 빠르게 발견할 수 있다.

백엔드가 문제를 수정하는 동안 모바일은 UI나 다른 기능을 구현할 수 있으므로, 양쪽이 병렬로 작업하고 출시 지연 위험을 줄일 수 있다.

## 4.2 아래 계층으로 작업하기

`CourseService`가 자신의 책임을 충분히 수행할 수 있을 때까지 플레이스홀더를 하위 계층으로 이동시킨다.

### 4.2.1 `CourseService` 완성하기

`CourseService`는 `Course`를 만들기 위해 다음 도메인의 데이터를 결합한다.

* `Tutor`
* `TodoList`
* `Calendar`

따라서 각 데이터를 불러오는 타입을 만들고, `CourseService`가 직접 반환하던 플레이스홀더를 해당 타입의 호출로 교체한다.

```text
기존
CourseService → Course.placeholder

변경
CourseService
├── TutorAPI → Tutor.placeholder
├── TodoAPI → TodoItem.placeholder
└── Calendar → CalendarEvent.placeholder
```

**플레이스홀더를 제거하는 것이 아니라 한 계층 아래로 이동시키는 것이다.**

하위 타입이 실제 데이터와 플레이스홀더 중 무엇을 반환하는지는 `CourseService`가 알 필요가 없다. `CourseService`가 데이터를 요청하고 조합하는 책임을 올바르게 수행한다면 현재 단계에서는 충분히 완성된 것으로 볼 수 있다.

### 4.2.2 새로운 타입 정의하기

네트워킹 방식이 아직 정해지지 않았다면 새 타입에도 플레이스홀더를 사용한다.

실제 비동기 환경을 어느 정도 재현하기 위해 플레이스홀더에 지연을 추가할 수도 있다.

### 4.2.3 더 깊은 계층으로 내려가기

다음에는 `TutorAPI`가 `NetworkClient`를 통해 데이터를 요청하고, 받은 데이터를 `parseTutor(from:)`에서 변환하도록 만든다.

이때 플레이스홀더는 `TutorAPI`에서 파싱 함수로 이동한다.

```text
CourseService
    ↓
TutorAPI
    ↓
NetworkClient
    ↓
parseTutor()
```

이처럼 큰 문제를 요청, 파싱, 저장과 같은 더 작은 문제로 계속 나눈다.

### 4.2.4 최종 결과

실제 백엔드 통합이 끝날 때까지 플레이스홀더를 계속 하위 계층으로 이동시킨다.

통합이 완료되면 다음 작업으로 넘어간다.

* 테스트를 추가해 모델 계층 성숙시키기
* 실제 UI 구현하기

### 4.2.5 백엔드도 플레이스홀더를 사용한다

백엔드가 아직 완성되지 않았더라도 고정된 JSON을 반환하는 API를 먼저 제공할 수 있다.

가짜 데이터라도 API 요청과 응답 형식을 먼저 연결하면, 백엔드가 이후 실제 데이터로 교체해도 앱은 동일한 방식으로 동작한다.

이는 **세부 구현보다 전체 시스템의 연결을 먼저 완성하는 방식**이다.

## 4.3 API 설계 후 메서드 최적화하기

API 경계를 먼저 안정시키면 호출부를 변경하지 않고 내부 구현을 개선할 수 있다.

예를 들어 처음에는 `Tutor`, `TodoList`, `Calendar`를 순차적으로 불러오고, 이후 `async let`을 사용해 병렬로 전환할 수 있다.

핵심은 **처음부터 최적화하지 않고, 먼저 전체 연결을 확인한 뒤 내부를 개선하는 것**이다.

---

## 4.4 전체론적 개발 돌아보기

전체론적 개발에서는 실제 입출력이 완성되지 않았더라도 다음을 먼저 확보할 수 있다.

* 전체 시스템의 구조
* 각 컴포넌트의 책임과 API
* 실행 가능한 코드
* UI를 연결할 수 있는 출발점
* 팀원에게 나눌 수 있는 작업 경계

주요 장점은 다음과 같다.

* 구조를 빠르게 변경할 수 있다.
* 구현 방법을 몰라도 플레이스홀더로 계속 진행할 수 있다.
* 매 단계에서 가장 중요한 작업을 다시 선택할 수 있다.
* API 경계를 기준으로 팀 작업을 나눌 수 있다.

따라서 전체론적 개발은 **불확실성이 많거나 새로운 코드베이스를 만드는 상황에 적합하다.**

### 4.4.1 구현 방법을 몰라도 진행할 수 있다

플레이스홀더로 API를 먼저 정의하면 구체적인 구현을 뒤로 미루고 다른 영역의 개발을 계속할 수 있다.

필요한 시점에 플레이스홀더를 실제 구현으로 교체하면 된다.

### 4.4.2 구조 변경이 가볍다

초기 구현에 많은 시간을 투자하지 않았기 때문에 컴포넌트의 위치나 책임을 비교적 쉽게 바꿀 수 있다.

예를 들어 `Store`를 `CourseService`, `TodoList`, `NetworkClient` 중 더 적절한 위치로 옮길 수 있다.

### 4.4.3 컨텍스트 전환과 업무 위임

단점은 여러 영역을 오가며 작업하므로 각 컴포넌트가 오랫동안 미완성 상태로 남는다는 것이다.

반면 팀에서는 API 경계를 기준으로 각 컴포넌트를 나누어 병렬로 개발할 수 있다.

### 4.4.4 하향식과 상향식

전체론적 개발은 기본적으로 상위 기능에서 시작해 하위 계층으로 내려가는 하향식 접근이다.

다만 시스템 전체가 특정 기술의 가능 여부에 의존한다면, 먼저 해당 기술을 검증하는 상향식 접근이 더 적절할 수 있다.

### 4.4.5 테스트 작성을 미룬다

초기에는 코드 위치, 이름, 책임과 구조가 자주 바뀐다.

이때 테스트까지 함께 작성하면 수정 비용이 커질 수 있으므로, 구조가 어느 정도 안정된 뒤 중요한 영역부터 테스트한다.

### 4.4.6 인터페이스부터 설계하지 않는다

모든 구체 타입에 대응하는 인터페이스를 만들면 타입 수와 복잡성이 불필요하게 증가한다.

따라서 **구체 타입과 API를 먼저 만들고, 구현 교체나 다형성이 실제로 필요할 때 인터페이스를 도입한다.**

테스트만을 위해 모든 타입에 인터페이스를 추가하는 것도 피한다.
