## 도입 — 광택보다 통합

에피그래프: *"광택보다 통합을 선호하라 — 작동하지만 못생긴 기능이 깨진 반짝이는 기능을 이긴다."*

지난 챕터(§3)에서 데이터 모델·기본 Store·작동하는 핵심 로직을 갖춘, **컴파일되고 실행되는 CourseService**를 만들었다. 이제 고전적 갈림길에 선다 — UI를 광택내 보여줄 것을 만들 것인가, 더 깊이 내려가 실제 백엔드에 연결할 것인가. 두 방향 모두 장점과 함정이 있다.

이 챕터는 두 경로를 모두 탐구한 뒤, 의존성 계층에서 **더 깊이** 작업하며 홀리스틱 개발을 계속한다. 모든 것이 연결되고 컴파일되도록 유지하면서 플레이스홀더를 실제 구현으로 하나씩 교체한다. 끝에 가서는 실제 네트워크 요청 직전까지 도달하고 거기서 멈추는데, 이 지점이 다음에 다룰 **의존성 주입과 테스트**로 점프하기에 좋은 출발점이기 때문이다. 마지막으로 홀리스틱 접근법의 트레이드오프를 회고한다 — **은총탄은 없다.**

## § 4.1 UI vs 다른 구현 — 두 방향의 비용과 이득

랜드스케이프를 보면 두 방향이 있다. **더 깊이(down)** 가서 플레이스홀더를 제거하고 백엔드 통합의 지저분한 현실에 가까워지거나, **더 높이(up)** 가서 Course를 실제 UI에 연결하거나.

**§ 4.1.1 UI에 집중하기 — 매력과 함정.** UI 경로는 Course 컴포넌트를 실제 뷰에 연결하는 것으로, 처음엔 라벨과 버튼뿐인 뼈대라도 **UI와 데이터가 어떻게 함께 작동할지 알아내는 구체적인 방법**이다. 그래프에서는 Course 위에 CourseUI 도메인(아직 미확정이라 점선)이 얹힌다(Figure 4.1). 매력은 분명하다 — 빠르게 보여줄 것이 생기고, 디자이너와 협업하기 좋고, 사용자 테스트·내비게이션 흐름 연결로 가는 가장 빠른 경로다. **사람들이 UI 완성을 보고 얼마나 흥분하는지 과소평가하지 마라** — 기술·비기술 인원 모두 진척과 모멘텀을 체감하고, 빌드를 배포해 손에 쥐어줄 수도 있고, 제품 데모 비디오로 마케팅에도 쓸 수 있다.

반면 UI 표면 아래 작업(데이터·파싱·백엔드)은 **덜 가시적**이다. 사람들은 티켓이 움직이는 것에 만족하지만 무슨 작업인지 항상 이해하지는 않는다 — 그들에게는 더 블랙박스다. 여기서 나오는 심리적 함정: UI로 시작하면 앱이 **초기에 꽤 완성된 것처럼 느껴지고**, 그 뒤로 비개발자에게 숨겨진 보이지 않는 작업이 산더미로 남는다. 이는 "처음엔 정말 빠르다가 나중에 느려지는 개발자"라는 **비뚤어진 시각**을 낳는다. 처방은 **모든 부분이 자리 잡을 때까지 광택 있는 UI 전달을 지연**하는 것 — 화려한 애니메이션 대신 플레이스홀더 UI를 만들어 실제 데이터에 연결한다.

UI 우선의 더 큰 위험은 **안전한 거품 안에서 좋아 보이는 UI를 만들고, 다 끝난 뒤 네트워크 구현을 시작해서야 팀에 심각한 정렬 오류(misalignment)가 있음을 꽤 늦게 깨닫는 것**이다. 통합을 일찍 시작하면 필요할 때 **일찍 경보를 울릴 수 있고**, 내가 다른 일을 하는 동안 팀원을 "활성화"해 중요한 업데이트를 하게 만들어 프로세스 전체의 시간을 절약한다.

**§ 4.1.2 더 깊이 들어가기 — 현실이 다가온다.** 다른 옵션은 HDD를 계속해 Tutor·Calendar·TodoList가 쓰는 API 구현 코드를 쓰고, 그것들이 다시 NetworkClient로 실제 네트워크 호출을 하게 만드는 것이다(Figure 4.2). 깊이 가는 것은 **시스템 통합에 집중하고 숨겨진 이슈를 발견하는** 훌륭한 옵션이다 — 백엔드 엔지니어와의 정렬, 일치하지 않는 오류 코드, 오래됐거나 아예 없는 백엔드 문서, 최신이 아닌 스테이징 환경, 서드파티 분석 라이브러리 토큰 유료 결제 필요 같은 것들이 여기서 튀어나온다.

통합 이슈는 어떤 각도에서든 온다 — 백엔드 엔지니어가 잘못된 필드를 구현했거나, 디자이너 의도와 다르게 필드가 옵셔널이라 그대로 두면 UI에까지 반영되거나, 알림에 대한 광범위한 논의(전화 토큰 제출용 새 엔드포인트가 필요하다는)를 촉발하거나. 심지어 **알림이 애초에 작동하지 않는다는 것을 동료가 신혼여행 떠나기 전에 알려줄 수도 있다.** 무엇이 됐든 백엔드와 통합하면 현실이 다가오고, **예상치 못한 이슈는 늦게보다 일찍 발견하는 것이 낫다.**

**§ 4.1.3 결론 — 백엔드 통합을 우선하라.** UI를 만들고 광택낸 뒤 통합을 시작하면 **가장 가능성 높게 릴리스가 지연된다**(Figure 4.3). 반대로 통합을 가능한 일찍 시작하면 나와 백엔드 개발자 모두 잘못된 가정이 어디 있는지 배우고, **이번에는 백엔드가 끝날 때까지 기다릴 필요가 없다 — 할 UI 작업이 충분히 있기 때문**이다. 백엔드에 관한 가정 위에서 일하지 않았으므로 앱에 다시 해야 할 작업도 거의 없다(Figure 4.4). 두 사람이 병렬로 일하고 릴리스 날짜가 위험해지지 않는다 — 이전의 직렬 워크플로우가 지연을 만들던 것과 대조된다.

## § 4.2 아래로 내려가기 — 플레이스홀더를 한 계층씩 밀어내기

방향이 정해졌으니 §3의 구현을 이어간다. 현재 `CourseService.loadFresh(id:)`는 `Task.sleep(2초)` 후 `Course.placeholder`를 반환할 뿐 **실제 네트워크 호출조차 하지 않는다**. 여기서부터 플레이스홀더를 실제 타입으로 대체해 나간다.

**§ 4.2.1 CourseService 마무리 — 세 새 타입 도입.** Course를 채우려면 Tutor·Calendar·TodoList 세 도메인의 데이터를 결합해야 하고, 각 도메인에는 로드할 타입이 필요하다. 그래서 각 도메인을 대표하는 **`TutorAPI`, `TodoAPI`, `Calendar`** 세 타입에 의존하기로 결정한다(Figure 4.6). 이들을 CourseService의 속성으로 추가하고 `loadFresh` 안의 플레이스홀더 코드를 세 타입의 실제 로딩 메서드 호출로 교체한다.

```swift
// Listing 4.2 — CourseService 안의 플레이스홀더 제거
final class CourseService {
    // 이 타입들은 아직 존재하지 않으니 컴파일러가 화낼 것이다!
    let tutorAPI = TutorAPI()
    let todoAPI = TodoAPI()
    let calendar = Calendar()

    func loadFresh(id: UUID) async throws -> Course {
        let tutor = try await tutorAPI.loadTutor(courseId: id)
        let schedule = try await todoAPI.loadSchedule(courseId: id)
        let calendarEvent = try await calendar.loadEvent(courseId: id)
        return Course(id: id, tutor: tutor, schedule: schedule, calendarEvent: calendarEvent)
    }
}
```

당연히 세 타입이 아직 없어 컴파일되지 않는다. 하지만 이제 무엇을 할지 안다 — 정의하고 플레이스홀더로 다시 컴파일되게 만든다. **효과적으로 플레이스홀더를 한 단계 더 깊이 밀어내고 있는 것이다.**

> **이 절의 중요한 깨달음** — *"CourseService에서 모든 플레이스홀더를 제거했으므로, 이 단계에서 그것을 끝났다고 간주할 수 있다!"* CourseService는 "올바른" 방식으로 데이터를 로드한다. 자신이 실제 네트워크 호출을 쓰지 않는다는 걸 인식하지 못하지만 **CourseService 자체는 완벽하게 괜찮다.** 새 의존성 셋이 미완성인 것은 우리에게 명백하지만, 그 셋이 실제 데이터를 로드하는지 플레이스홀더를 로드하는지는 **CourseService에 전혀 중요하지 않다.**

**§ 4.2.2 새 타입 정의 — 지연을 곁들인 플레이스홀더, 그리고 `final`.** 세 타입을 다 완성하는 건 작업이 너무 많고 네트워킹도 아직 미정이므로 플레이스홀더 구현을 쓴다. 이전과 마찬가지로 **실제 동작을 모방하기 위해 각 플레이스홀더 앞에 지연을 넣는다**(§3.5의 인공 지연 원칙 반복).

```swift
// Listing 4.3
final class TutorAPI {
    func loadTutor(courseId: UUID) async throws -> Tutor {
        try await Task.sleep(for: .seconds(2))
        return Tutor.placeholder
    }
}
final class TodoAPI {
    func loadSchedule(courseId: UUID) async throws -> [TodoItem] {
        try await Task.sleep(for: .seconds(2))
        return Array(repeating: TodoItem.placeholder, count: 10)
    }
}
final class Calendar { /* loadEvent → CalendarEvent.placeholder */ }
```

이제 모든 것이 다시 컴파일되고, **CourseService는 끝났다고 말할 수 있다.** 이 프로세스를 반복해 모든 도메인을 끝낼 수 있다.

> **왜 모든 클래스를 `final`로 만드는가** — 다른 사람이 우리 클래스를 서브클래싱하는 것을 막기 때문이다. 우리는 지금(또는 영원히) 서브클래싱을 지원할 의도가 없다. 그렇지 않으면 누군가 우리 클래스의 서브클래싱에 의존하게 되고, 그걸 **우연히 깨뜨릴 수 있다**. (Kotlin은 클래스가 기본 `final`이라 이 습관이 언어에 내장돼 있다.)

**§ 4.2.3 프로세스 계속 — NetworkClient에 같은 패턴을 한 번 더.** 세 새 타입은 네트워크 요청 설정과 원시 데이터 파싱을 담당하며, 이를 새 클래스 **`NetworkClient`**를 통해 달성한다(Figure 4.7). NetworkClient의 목표는 **비즈니스 로직을 인식하지 않으면서 네트워크 호출에만 집중**하는 것이다.

TutorAPI에 집중하면 세 단계가 그대로 반복된다:

| 단계 | 무슨 일이 일어나는가 |
|---|---|
| ① 플레이스홀더를 한 칸 아래로 | `loadTutor`는 `networkClient.request(endpoint:)`로 `Data`를 받고, **플레이스홀더는 TutorAPI 내부의 private `parseTutor(from:)`으로 옮겨간다** — 파싱은 NetworkClient가 아니라 TutorAPI의 책임 |
| ② 다시 컴파일 실패 | `NetworkClient` 타입이 아직 존재하지 않기 때문 |
| ③ 플레이스홀더로 정의 | `request(endpoint:)`가 `Task.sleep(2초)` 후 **빈 `Data()`** 를 반환하도록 정의해 다시 컴파일되게 만든다 |

```swift
// Listing 4.4 — NetworkClient 의존성으로 TutorAPI 구현
final class TutorAPI {
    let networkClient = NetworkClient()

    func loadTutor(courseId: UUID) async throws -> Tutor {
        let data = try await networkClient.request(endpoint: "/api/courses/\(courseId)/tutor")
        return try parseTutor(from: data)   // 플레이스홀더가 여기로 내려왔다
    }

    private func parseTutor(from data: Data) throws -> Tutor {
        return Tutor.placeholder            // 실제 JSON 파싱은 아직
    }
}
```

핵심은 **`loadTutor(courseId:)`를 이 단계에서 "충분히 끝났다"고 간주할 수 있다**는 것이다 — "요청을 어떻게 만들고 파싱할지" 두 문제를 안고 있던 함수가 이제 **파싱 하나만 남은 문제로 줄었기** 때문이다.

**§ 4.2.4 결과 — 어디까지 왔는가.** 계층에서 꽤 깊이 내려왔다. 이 시점의 상태는 다음과 같다.

- **실제 구현으로 작동**: Course · Tutor · Calendar · TodoList 도메인
- **여전히 꽤 정의되지 않음**: Store · Networking 도메인
- **권장 다음 행보**: 적절한 백엔드 통합으로 끝날 때까지 **플레이스홀더를 계속 제거해 나가는 것**

궁극적으로 실제 NetworkClient로 돌아가는 진짜 애플리케이션이 되겠지만, 저자는 **그 부분을 독자를 위한 연습 문제로 남긴다** — 대부분의 모바일 앱에서 볼 수 있는 단도직입적인 네트워크 호출일 뿐 화려한 것이 없기 때문이다. 대신 **NetworkClient와 관련된 진짜 복잡한 작업은 의존성 주입과 테스트**이며, 그래서 다음 챕터가 정확히 거기에 집중한다. 백엔드 통합이 끝난 뒤에는 (테스트를 추가해) 모델 계층을 성숙시키거나 UI로 올라갈 것을 제안한다.

**§ 4.2.5 백엔드도 플레이스홀더를 써야 한다 — 앱 경계를 넘는 HDD.** 같은 원리가 백엔드에도 적용된다. 백엔드 개발자가 막 시작해 **데이터베이스조차 없더라도**, 최소한 **고정 JSON 파일만 반환하는 플레이스홀더 API 엔드포인트**를 제공하게 하라. JSON이 가짜 데이터여도 앱은 API 호출 통합을 시작할 수 있고, 나중에 백엔드가 그 고정 JSON을 라이브 데이터로 바꿔도 **앱은 차이를 알아채지조차 못한다.** 이는 통합을 훨씬 일찍 시작하게 해주며(§4.1.3의 이득), 정확한 디테일을 채우기 전에 모든 것을 연결한다는 홀리스틱 방식이 **앱 경계를 가로질러서도** 성립함을 보여준다.

## § 4.3 API 설계 후 메서드 최적화 — 안정된 경계 뒤에서의 자유

API를 적절히 설계해 두면 **다른 타입에 영향을 주거나 리팩터링할 필요 없이 함수 본체 안에서 마음껏 반복할 자유**를 얻는다. 그 증거로 CourseService를 개선한다 — 현재 tutor·schedule·calendarEvent를 하나씩 직렬로 로드하고 있으니, 셋을 동시에 시작하고 **동기화 지점(synchronization point)** 에서 모두 끝나기를 기다린 뒤 Course를 생성하도록 바꾼다.

```swift
// Listing 4.7 — 병렬 호출로 loadFresh 최적화
final class CourseService: Sendable {
    func loadFresh(id: UUID) async throws -> Course {
        async let tutor = try tutorAPI.loadTutor(courseId: id)          // 병렬 시작
        async let schedule = try todoAPI.loadSchedule(courseId: id)
        async let calendarEvent = try calendar.loadEvent(courseId: id)
        return try await Course(id: id, tutor: tutor,                    // ← 동기화 지점
                                schedule: schedule, calendarEvent: calendarEvent)
    }
}
```

각 `await`를 `async let`으로 바꾸고 Course 생성 지점을 `await`로 표시한다. (Swift 한정 디테일 — `async let`이 병렬 작업에서 동시에 `self`에 접근하므로 CourseService와 그 의존성을 `Sendable`에 따르게 만들어 동시성 경계를 넘어도 안전하다고 알린다.)

**여기서 중요한 건 문법이 아니라 이 최적화가 호출부를 한 줄도 건드리지 않았다는 사실이다.** 저자의 결론: ***"구현 디테일에 너무 집중하기 전에 사물이 어떻게 연결되는지에 집중하라. 충분히 작동하게 만들고, 그 후 개선하라."***

## § 4.4 홀리스틱 주도 개발 회고 — 은총탄은 없다

**§ 4.4 도입 — 지금 손에 쥔 것.** 작동하는 프로그램이 있다. I/O 부분만 아직 작동하지 않을 뿐 — 실제 서버에서 데이터를 가져오지도, 파일 저장소를 쓰지도 않는다. **그런데 그것이 바로 이 방식의 아름다움이다.** 단순한 솔루션이지만 이미 무엇을 만들지 알아냈고, 일부 구현을 했고, 작업을 팀원끼리 나눌 수 있고, 누군가에게 UI 연결을 맡길 수 있다. 강하게 분리된 도메인 덕분에 코드를 그대로 두거나, 자체 저장소로 옮기거나, 오픈 소스로 만들거나, 로컬 모듈로 만들 유연성도 있다. **선을 그었고, 이제 색칠을 시작할 수 있다.**

진짜 이점은 두 가지다 — **시간을 크게 희생하지 않고 빠르게 리팩터링하고 심지어 코드를 삭제할 수 있다**는 것, 그리고 더 크게는 **가장 높은 우선순위 작업에 집중할 수 있다**는 것. Store를 갖는 것은 높은 우선순위지만, **지름길을 택해 메모리 저장소로 만드는 것이 그 우선순위 자체를 낮춘다**(§3.6의 순진한 Store가 여기서 정당화된다). 매 단계 후 "가장 중요한 것은 무엇인가?"를 확인하고 다음으로 넘어간다.

**§ 4.4.1 HDD는 진행할 자신감을 준다.** 구현 디테일을 **모른 채로도** 진행할 수 있다. 로컬 저장소용 MySQL을 구현해야 하는데 해본 적이 없다고 하자 — 플레이스홀더 구현으로 API를 먼저 정의해 "실제" 구현을 지연시키고, 그동안 자기 여유에 맞춰 MySQL을 공부하면 실제 구현 시점에 훨씬 잘 준비돼 있다. 때가 되면 함수 본체만 실제 MySQL 저장소로 교체하고, 인터페이스를 약간 조정하면 끝이다.

**§ 4.4.2 경량 재구조화.** 도메인과 컴포넌트 계층으로 사고하므로 **컴포넌트를 사방으로 던지는 일이 가벼워진다.** Store 위치가 잘못됐다고 판단할 수도 있다 — CourseService가 Store를 알 필요가 없다거나, TodoList 데이터만 저장하는 게 낫다거나, 더 극단적으로 **모든 네트워크 호출을 캐시하도록 NetworkClient가 내부적으로 Store를 쓰게** 만들 수도 있다. 문제되지 않는다. Store는 독립형 컴포넌트라 어디에든 살 수 있고, **구현에 시간을 많이 잃지 않았으므로 결정을 되돌리는 것이 고통스럽지 않다.**

**§ 4.4.3 컨텍스트 스위칭과 위임 — 단점, 그리고 그 단점의 전환.** HDD의 단점은 한 컴포넌트를 제대로 끝내는 대신 **모든 곳을 점프하며 모든 컴포넌트를 조금씩 끝낸다**는 것이다. 모든 컴포넌트가 훨씬 오래 미완성·미해결 상태로 머물고, 정신적으로는 **아무것도 정말 해결되지 않았기 때문에 뇌에서 더 많은 공간을 차지**한다.

> 저자의 비유 — 새 집에 이사 와서 한 방 벽을 다시 칠하지만 바닥은 안 깔고, 욕실 타일을 다시 깔지만 싱크대는 설치하지 않고, 새 부엌을 만들지만 수도 라인에 연결하지 않은 상태. **아무것도 정말 "끝나지" 않았다** — 모든 것이 영구적으로 반쯤 완료된 채 살기에 짜증나는 곳.

이 미완성 상태는 **줌인해서 핵심 조각에 집중하고, 머릿속에서 끝내고, 잊고 넘어가고 싶어하는 사람들에게 특히 어렵다.** 그러나 팀 환경에서는 이것을 **이점으로 전환**할 수 있다. 전체 랜드스케이프를 감독하는 리드 엔지니어가 있다면, 단일 부분에 집중하기를 선호하는 사람들에게 더 작은 도메인/컴포넌트를 **위임**하면 된다 — 한 사람은 Store, 다른 사람은 TodoList, 또 다른 사람은 CourseUI. **그중 누구도 작업을 위해 전체 시스템을 알 필요가 없다.** API에 집중함으로써 팀 리드는 각 부분이 작업되는 동안에도 서로 연결되도록 보장한다. 그래서 홀리스틱 접근법은 **팀 환경에서 훨씬 빠를 수 있고 스케일링된다.**

**§ 4.4.4 톱다운 vs 보텀업 — HDD가 맞지 않는 경우.** 홀리스틱 개발은 **기본적으로 톱다운 접근법**이며, 그래서 맞지 않을 수 있다. 랜드스케이프 위에서 시작해 작동할 때까지 아래 계층으로 내려가는 방식인데, **의존하려는 핵심 기술이 실제 사용 사례에 통할지 확신할 수 없을 때는 바닥에서 시작하는 게 낫다.** 저자의 예: 전체 앱을 백그라운드에서 풀타임 실행한다는 전제 위에 기반을 두었는데 **OS가 그것을 허용하지 않는 경우** — 다른 도메인을 구현하기 전에 그 한계를 자세히 배우고 우회 방법부터 알아내야 한다.

**§ 4.4.5 테스트 작성을 지연하는 이유.** 우리 접근법은 **TDD와 비슷하지만 테스트가 없다.** 더 나쁘게 들리는가? TDD도 매 단계 진행하기에 충분한 코드만 쓴다는 점은 같고, 다른 점은 TDD가 즉시 테스트한다는 것이다. 아직 테스트를 쓰지 않는 이유는 두 가지다:

1. **코드가 계속 움직인다** — 필요한 부분을 끊임없이 알아내고, 옮기고, 이름을 바꾸고, 삭제하는 중이다. 프로덕션 코드에 더해 테스트 코드까지 리팩터링해야 하면 느려지고, **곧 삭제할 테스트를 쓰는 것은 시간 낭비**다.
2. **특정 영역만 테스트할 것이기 때문** — 모든 것을 테스트 가능하게 만드는 것을 피한다. 그러지 않으면 **인터페이스가 사방에 널린 결과**로 끝날 위험이 있다. (저자도 논쟁의 여지가 있는 주제임을 인정하며 테스트 챕터로 미룬다.)

이제 구조 — **어쩌면 아키텍처라 부를 수도 있는 것** — 가 자리 잡았으므로, 의존성과 테스트를 생각하기에 더 나은 위치에 있다.

**§ 4.4.6 왜 인터페이스로 설계하지 않는가.** 구체 타입의 API에 그렇게 집중한다면 아예 인터페이스(프로토콜)를 쓰지 그러냐는 질문이 나온다. 저자의 답변:

| 반론 | 내용 |
|---|---|
| **인터페이스만으론 프로그램이 없다** | 인터페이스 타입을 써도 **여전히 구체 구현이 필요하다.** 인터페이스만 정의하면 프로그램이 아니다. 구체 타입이 필요하니 거기 먼저 집중한다 — **인터페이스 타입은 사전이 아니라 필요할 때 자연스럽게 온다.** |
| **타입이 두 배가 되고 추론이 어려워진다** | 프로토콜 지향 프로그래밍은 가능하지만 **코드베이스 이해 난이도를 극적으로 올린다.** 모든 타입이 두 배가 되고, 메서드가 초기 이점 없는 프로토콜에 살게 돼 **메서드를 찾기도, 어떤 타입이 어떤 구현을 오버라이드하는지 추론하기도 더 어려워진다.** 용도는 분명 있지만 작고 신선한 코드베이스에서는 아니다. |
| **다형성이 필요 없다** | 인터페이스를 쓰는 주된 이유는 교체 가능한 다형성인데, 우리는 **TodoAPI도 하나, CourseService도 하나**다. `TodoListLoader`, `CourseServiceProtocol` 같은 걸 왜 추가하나 — 복잡도만 늘 뿐이다. |
| **이름을 못 짓는 것 자체가 신호다** | 저자 자신을 포함해 사람들이 이런 **"미러(mirror)" 인터페이스의 이름 짓기를 어려워하는데, 그것이 코드 스멜을 나타낸다.** |
| **"테스트 때문에"도 스멜이다** | *"인터페이스를 쓰면 훨씬 테스트 가능해진다"* — **거기에 범인이 있다.** 모든 타입이 테스트를 위해 인터페이스를 필요로 한다고 생각하는 것 자체가 스멜이며, 단지 테스트를 위해 프로덕션 코드베이스를 **더 다공성(porous)으로** 만드는 셈이다. 테스트 챕터에서 **더 적은 인터페이스로도 높은 커버리지가 가능함**을 보인다. |

## § 4.5 챕터 요약

| 축 | 핵심 |
|---|---|
| **전략적 개발 결정** | 빠른 데모·이해관계자 동의·사용자 테스트 피드백이 필요하면 UI 우선. 그러나 백엔드 정렬 오류·스테이징 이슈·팀 조정 문제를 일찍 발견하려면 **통합 우선을 선호**하라. 진척에 대한 잘못된 인상을 피하려 **보이지 않는 작업이 끝날 때까지 광택 UI를 지연**하라. 병렬 개발과 머지 충돌 감소를 위해 **API 경계로 팀 작업을 나눠라** |
| **홀리스틱 구현 심화** | 모멘텀 유지를 위해 **한 번에 한 추상화 계층씩** 플레이스홀더를 체계적으로 대체하라. 개별 컴포넌트를 완전히 구현하는 대신 **플레이스홀더를 계층 아래로 밀어내라**. **API가 안정되면 구현이 플레이스홀더라도 "충분히 끝났다"고 간주**하라. 최우선 도메인에 집중을 유지하려 **비즈니스 로직 → 인프라 방향으로** 작업하라 |
| **API 설계 후 최적화** | 안정적 API를 먼저 설계한 뒤 **호출부에 영향 없이 구현을 최적화**하라. **노출된 모든 요소가 유지보수 약속**이므로 API 표면을 작게 유지하라. 구현 변경을 소비자 코드에서 격리하는 데 API 경계를 써라 |
| **트레이드오프 관리** | 아키텍처 비전을 유지하는 대가로 **컨텍스트 스위칭과 정신적 오버헤드를 받아들여라**. 코드 구조가 변동성이 클 때는 **테스트 작성을 지연**하라. 비핵심 컴포넌트의 **인지된 우선순위를 낮추는 데 플레이스홀더를 써라**. 홀리스틱 접근법에서는 **어떤 타입도 처음부터 끝나지 않지만**, 누군가 아키텍처를 감독하는 한 동료들은 특정 컴포넌트에 집중할 수 있다 |

## 나누고 싶은 이야기

### 1. Compose/Kotlin으로 옮기면 — `async let`은 `async {}`, `final`은 기본값

- **`async let` 병렬화 → `coroutineScope { async { } }`.** Swift의 `async let` 3개 + 동기화 지점 `await` 패턴은 Kotlin에서 `coroutineScope { val tutor = async { tutorAPI.loadTutor(id) }; val schedule = async { ... }; val event = async { ... }; Course(id, tutor.await(), schedule.await(), event.await()) }`가 된다. 개념은 정확히 대응한다 — `async {}`가 병렬 시작, `.await()` 지점들이 동기화 지점. 그리고 **Swift의 `Sendable` 준수 요구는 Kotlin에서 대응물이 거의 없다** — 코루틴은 구조적 동시성과 `CoroutineScope`가 안전성을 담당하므로, 이 챕터에서 `Sendable`에 들인 지면은 안드로이드 관점에서 그냥 넘어가도 되는 Swift 전용 세금이다.
- **`final class` → Kotlin은 클래스가 기본 `final`.** §4.2.2에서 저자가 "들이면 좋은 습관"이라 말한 것이 Kotlin에서는 **언어가 강제하는 기본값**이다(열려면 명시적으로 `open`). 저자의 논거("의도치 않게 서브클래싱 계약을 만들지 마라")가 그대로 Kotlin 설계 철학의 근거이기도 하다.
- **플레이스홀더 지연 `Task.sleep(for:)` → `delay(2000)`.** §3과 동일.
- **`Data` 반환 NetworkClient → Retrofit/Ktor 이전 단계의 raw 계층.** §4.2.3에서 NetworkClient가 `Data`만 반환하고 파싱은 `TutorAPI.parseTutor(from:)`이 맡는 구조는, 안드로이드에서 **Retrofit 인터페이스가 `ResponseBody`/raw JSON을 주고 Mapper가 도메인 모델로 변환하는 계층 분리**와 같다. "네트워크 클라이언트는 비즈니스 로직을 인식하지 않는다"는 원칙이 곧 "Retrofit service는 DTO까지만, 도메인 매핑은 Repository/Mapper에서"라는 익숙한 규칙이다.
- **§4.4.6의 "인터페이스로 설계하지 마라"는 안드로이드 관습과 정면으로 충돌한다.** 안드로이드 진영은 `interface UserRepository` + `class UserRepositoryImpl`을 거의 반사적으로 만드는데, 저자는 정확히 이것을 **"미러 인터페이스 — 이름조차 못 짓는 코드 스멜"** 로 지목한다. `Impl` 접미사를 붙이는 순간 이름 짓기에 실패했다는 저자의 진단이 그대로 적중한다. Hilt로 `@Binds`를 쓰려면 인터페이스가 필요하다는 것이 관습의 실질적 이유인데, 저자는 **그 이유("테스트/DI 때문에")마저 스멜**이라고 본다 — 이 논쟁은 §7~9 DI 파트에서 다시 붙을 주제이니 판단은 유보하되, 지금은 **"인터페이스가 필요해서 만든 것인가, 습관이라 만든 것인가"** 를 자문하는 것만으로 충분하다.

### 2. Pillo 앱에 대입하면 — DrugBank·FDA 통합을 지금 시작해야 하는 이유

§4.1.3의 "통합 우선"은 Pillo처럼 **외부 API 세 개(DrugBank·FDA·Pillo 자체)에 의존하는 앱에서 훨씬 강한 주장**이 된다. 통합 지점이 하나뿐인 앱이라면 UI를 먼저 다듬어도 리스크가 작지만, 세 소스는 각각 필드 이름·옵셔널 여부·오류 코드·레이트 리밋·인증 방식이 다르고 **정렬 오류가 세 배로 튀어나올 수 있다.** 복약 UI를 예쁘게 완성한 뒤에야 "FDA 응답에 우리가 기대한 용량 필드가 없다"를 발견하면, §4.1.1이 경고한 **"안전한 거품 안에서 만든 UI + 늦게 발견한 정렬 오류"** 시나리오가 그대로 재현된다.

실무 순서로 옮기면: ① `DrugRepository`의 호출부 API를 먼저 확정하고(§3의 원칙), ② `DrugBankApi`/`FdaApi`/`PilloApi` 세 타입을 `delay` + `DrugInfo.placeholder` 반환으로 정의해 컴파일·실행 상태를 유지한 뒤, ③ **세 API 중 가장 불확실한 것부터** 실제 호출로 교체해 응답 스키마를 검증한다. §4.2.3의 계층 구조(NetworkClient는 raw만, 파싱은 각 API 타입 내부)를 따르면 세 소스의 파싱 차이가 각자의 API 클래스에 격리되고, **통합 모델(`DrugInfo`)로 정규화되는 지점이 한 군데로 모인다.**

§4.2.5(백엔드도 플레이스홀더)는 Pillo에서 **자체 백엔드에만 적용 가능한 조언**이라는 점도 짚어둘 만하다. DrugBank·FDA는 외부 서비스라 "고정 JSON을 반환하는 엔드포인트를 만들어달라"고 요청할 수 없다 — 대신 **실제 응답 샘플을 한 번 받아 저장해두고 그것을 플레이스홀더로 쓰는 것**이 등가 전략이다. 이 편이 오히려 낫다: 내가 상상한 스키마가 아니라 **실제 스키마**로 개발하게 되므로, 나중에 실제 호출로 바꿔도 §4.2.5가 약속한 대로 "앱이 차이를 알아채지 못한다".

§4.4.3의 위임 논리도 Pillo가 1인 개발이라면 그대로 적용되지는 않지만 뒤집으면 유효하다 — **혼자 개발할 때 HDD의 단점(모든 것이 반쯤 완료된 채 뇌를 점유)은 완화되지 않고 전부 내 몫이다.** 그래서 "다음에 뭐가 가장 중요한가"를 매 단계 명시적으로 적어두는 습관(§4.4 도입부)이 팀 개발보다 오히려 더 중요하다.

### 3. 오늘의 멘탈 모델 업데이트 — "끝났다"의 정의가 계층마다 다르다

이 챕터를 하나로 관통하는 문장은 §4.2.1의 **"CourseService에서 모든 플레이스홀더를 제거했으므로 끝났다고 간주할 수 있다"** 이다. 여기서 "끝났다"는 **전체 시스템이 실제로 동작한다는 뜻이 아니라, 이 타입이 자기 계층에서 할 일을 다 했다는 뜻**이다. CourseService는 세 의존성이 가짜 데이터를 주는지 모르고, 알 필요도 없다. 같은 판정이 §4.2.3의 `loadTutor`에도 반복된다 — 요청+파싱 두 문제를 안고 있던 함수가 파싱 하나만 남기고 나머지를 NetworkClient에 넘긴 순간 "충분히 끝났다"가 된다.

**퀴즈에서 반복해 걸렸던 지점들도 결국 이 프레임에서 재구성된다.**

- **§4.2.3 NetworkClient (5회 중 1회 정답)** — 이 절이 새로 가르치는 것은 없다. §4.2.2에서 TutorAPI/TodoAPI/Calendar에 했던 **똑같은 3단계(플레이스홀더를 아래로 밀기 → 컴파일 실패 → 플레이스홀더로 정의)를 NetworkClient에 한 번 더 반복**할 뿐이다. 헷갈렸던 세부 — 파싱은 NetworkClient가 아니라 TutorAPI의 private 함수로 내려간다, `request`는 빈 `Data()`를 반환한다, `Sendable`은 §4.3의 이야기고 CourseService 속성 추가는 §4.2.1의 이야기다 — 는 모두 "이 절은 반복이다"를 붙잡으면 따라 나온다.
- **§4.2.4 결과** — 이 시점에 **작동하는 것은 Course/Tutor/Calendar/TodoList, 미정의는 Store/Networking**. 그리고 진짜 애플리케이션 완성은 **저자가 직접 보여주지 않고 연습 문제로 남긴다** — 다음 챕터가 다루는 것은 네트워크 구현이 아니라 **DI와 테스트**다.
- **§4.4.4 톱다운 vs 보텀업** — HDD를 무조건 옳은 방법으로 외우면 이 절이 안 잡힌다. HDD는 **톱다운이라는 하나의 선택지**이고, **의존하려는 핵심 기술/OS 한계가 사용 사례에 통할지 모를 때는 보텀업이 낫다**는 것이 저자가 명시한 예외다.

§3이 "호출부 우선"이라는 하나의 규율이었다면, §4는 그 규율을 **아래 방향으로 반복 적용하는 절차**와 **그 절차의 대가**를 함께 다룬다. 매 반복은 항상 같은 모양이다 — 위 계층의 플레이스홀더를 지우면 컴파일이 깨지고, 깨진 자리에 새 타입을 플레이스홀더로 정의하면 다시 컴파일된다. 그리고 그때마다 위 계층은 "끝났다"가 된다.
