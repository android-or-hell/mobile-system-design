## 디자인 시스템이란

- 디자인 시스템은 재사용 가능한 컴포넌트와 가이드라인, 동일한 UX를 제공하는 UI 모음집이다.
- 디자인 시스템은 개발자, 기획자, 디자이너 등 관련자들 사이에 공식적으로 공유된 UI 모음집을 의미한다.

## 디자인 시스템과 UI 라이브러리

- UI 라이브러리는 디자인 시스템의 일부일 수도, 단독으로 존재할 수도 있다.
- UI 라이브러리는 보통 색상, 폰트, 아이콘, 재사용 가능한 컴포넌트를 포함한다.
   - 즉, 인터페이스의 시각 일관성 보장을 위해 **구현**에 주로 초점을 맞춘다.
- 디자인 시스템은 위의 것들을 포함할 뿐 아니라, 그것을 **언제, 어떻게 쓸지에 대한 가이드라인과 원칙을 정의**한다.
- UI 라이브러리는 보통 개발자가 만들고, 이는 디자이너가 개발자에게 의존하게 만든다.
   - 이로 인해 컴포넌트를 한 번 업데이트 하려면 개발자와 디자이너는 서로 논쟁하고 자기의 입장을 견고히 해야 한다.
   - 즉 불필요한 커뮤니케이션 비용이 생기는 셈이다.
- 디자인 시스템은 좀 더 형식적으로, 개발자가 아닌 팀에 의해 결정된다.
   - 디자인 시스템은 대화를 "컴포넌트가 그것을 염두에 두고 만들지 않았어" 에서, "일관성을 유지하면서 새 요구를 만족시킬 수 있도록 어떻게 이 컴포넌트를 진화시킬 수 있을까?" 로 전환한다.
- 결론적으로 두 시스템의 차이는, **UI 라이브러리의 SSOT 는 코드에, 디자인 시스템의 SSOT 는 문서**에 있다.

## 디자인 시스템은 왜 개발자에게 중요한가
- 디자인은 모든 인터렉션이나 동적 상태를 설명할 수 없다.
   - 디자인이 아무리 많은 화면을 제공해도 100% 완전할 수 없다.
- 디자인 시스템을 통해 개발자는 그 틈을 메울 수 있다.
   - 예를 들어, UI가 올바르게 설정되어 있다면 라이트 모드 버전이 구현되었을 때 개발자는 별도 컴포넌트없이 다크 모드 버전을 손쉽게 구현할 수 있다.
- 즉, 디자인 시스템이 있으면 개발자는 디자이너가 없어도 좀 더 능동적으로 화면을 구현할 수 있다.
- 또한 신규 개발자의 입장에서는 문서화된 디자인 시스템을 통해 더 빠르게 프로젝트에 기여할 수 있다.

## 디자인 시스템의 적용
- 디자인 시스템의 구현은 솔직히 쉽지 않다.
- 실용적 접근 방법은 UI 라이브러리부터 구현하는 것이다.
- 디자인 시스템의 핵심적인 부분이기 때문에, 추후 디자인 시스템으로 변경하더라도 좀 더 쉽게 대응이 가능하다.
