> Mobile System Design 시리즈 Book 2 (Scale) 의 첫 세 챕터.

## Scale-00 · Where We Left Off (Book 2 도입)

Book 2 는 개별 기능에서 큰 코드베이스와 팀 단위로 시선을 옮긴다. 축이 두 개다.

### 1. Scaling Up UI

UI 라이브러리는 이미 있다. 이제 질문은, 그걸 디자인 시스템으로 공식화할 것인가다. 답은 늘 명백하지 않다. 모든 팀이 풀 스케일 디자인 시스템을 필요로 하지는 않는다. 다만 다듬어진 UI 와 멀티플랫폼 요구가 커질수록 디자인 시스템의 필요도 같이 커진다.

디자인 시스템을 만들든 안 만들든, 가장 기초가 되는 영역(타이포그래피·아이콘·그림자·간격)에 집중해서 UI 라이브러리를 보강한다. 그 다음 단계가 *토큰*. 디자인 결정을 명명된 값으로 굳혀 플랫폼·앱·팀을 가로질러 UI 를 정렬한다.

### 2. Scaling Up the Entire App

UI 는 방정식의 한 부분이다. 이제 초점은 전체 애플리케이션 아키텍처로 옮겨간다. *모놀리식 코드베이스* 를 모듈화되고 유지보수 가능한 시스템으로 바꾸는 작업. 모듈을 추출하고, 인터페이스를 설계하고, 모듈의 역할을 정하고, 모듈·팀 사이에 책임을 나누는 일. 목표는 속도·안정성·유연성의 균형을 잡으면서 더 큰 조직과 더 넓은 코드베이스를 받쳐주는 것이다.

### § 1.2 디자인 시스템이란 무엇인가

먼저 헷갈리기 쉬운 구분 하나. *시스템 디자인* (기술 아키텍처·엔지니어링) 과는 다르다. *디자인 시스템* 은 사용자 인터페이스에 초점이 있다.

공식 정의는 이렇다. *"제품과 플랫폼 전반에 걸쳐 통일되고 일관된 사용자 경험을 보장하는, 재사용 가능한 컴포넌트, 가이드라인, 표준의 모음."* 더 간단히 말하면, 제품과 플랫폼 전반에 걸쳐 일관된 룩 앤 필을 만들어 주는 장치다.

형태는 다양하다. 본격적인 웹사이트일 필요는 없다. 위키, README, Figma 라이브러리, Sketch 라이브러리도 모두 디자인 시스템이다. 모바일에선 SwiftUI/Jetpack Compose 의 재사용 컴포넌트, 코드베이스에 빌트인된 중앙화 스타일, 공유 폴더의 디자인 사양 같은 형태로 나타난다.

---

### § 1.3 디자인 시스템 vs UI 라이브러리

한 줄로 잡으면, UI 라이브러리는 *"무엇(what)"* 을, 디자인 시스템은 *"왜(why)·어떻게(how)"* 를 다룬다.

| | UI 라이브러리 | 디자인 시스템 |
|---|---|---|
| **제공물** | 컴포넌트의 *구현* | 컴포넌트 + *언제·어떻게* 의 가이드라인 |
| **예** | 재사용 가능한 *버튼* | primary vs secondary 사용 시점, 접근성, 시각 위계, 적절한 간격 |
| **소유권** | *개발자* (코드 = 진실의 원천) | *팀들* (공유 모델 — 디자이너·개발자 공동) |
| **진실의 원천 위치** | *코드* | *문서* |

소유권 차이가 중요한 이유는 이렇다. UI 라이브러리는 디자이너가 변경을 원해도 *"이 컴포넌트는 그것을 염두에 두고 만들지 않았어요"* 같은 마찰을 일으킨다. 디자인 시스템은 대화를 *"어떻게 진화시킬 수 있을까?"* 쪽으로 돌려놓는다.

> UI 라이브러리는 단독으로 존재할 수도 있고, 디자인 시스템의 일부일 수도 있다. (Figure 1.1 의 벤다이어그램)

### § 1.4 개발자에게 왜 중요한가 — 6가지 이점

> *"디자이너의 시간을 줄이는 도구가 아니라 개발자 자신을 위한 도구."*

첫째, 협업 문화가 달라진다. 디자이너가 UI 게이트키퍼가 되는 흐름이 끊기고, UI 책임을 개발자가 같이 진다. 핸드오프도 완성된 청사진이 아니라 출발점으로 보게 된다.

둘째, 무언의 결정을 채워준다. 디자인은 100% 완전할 수 없다. RTL, 다양한 콘텐츠 길이, 폰트 크기, 동적 상태까지 다 못 잡는다. 디자인 시스템이 그 틈을 메우는 원칙을 준다.

셋째, 개발자가 자율적으로 움직일 수 있다. 디자이너의 결정을 기다리지 않고 바로 구현에 들어간다.

넷째, 타 플랫폼 기능을 적응할 때 빌딩 블록, 규칙, 패턴이 준비돼 있다. iOS → Android 또는 그 반대 번역에서 또 한 번 미팅을 잡지 않고도 디테일을 처리할 수 있다.

다섯째, 숙련 개발자에게도 가치가 있다. 능숙해도 작은 불일치는 스며든다. 능숙한 엔지니어가 많다고 잘 소통한다는 보장은 없다. 디자인 시스템은 분리된 UI 가 만들어지는 위험을 줄인다.

여섯째, 신규 입사자 온보딩이 빨라진다. 비슷한 컴포넌트를 찾으려고 코드베이스를 뒤지거나 하드코딩할 필요가 없다. 미리 만들어진 컴포넌트와 레이아웃 패턴을 문서와 함께 참조하면 된다.

### § 1.5 도입의 도전

디자인 시스템은 단순한 컴포넌트 모음이 아니다. Figure 1.2 에 따르면 다음을 포함한다.

- **UI 라이브러리** (다중 구현)
- **토큰** (디자인 결정의 표현)
- **문서** (모두가 사랑한다는…)
- **디자인 자산** (아이콘·컴포넌트·템플릿·로고·일러스트레이션)
- **거버넌스** (디자인 시스템을 보호하는 규칙·프로세스)
- **툴링 & 자동화** (디자이너·개발자 삶을 쉽게)

마이그레이션이 까다로운 이유. 팀들이 기능 마감일을 디자인 시스템 마이그레이션보다 앞에 둔다. *"빅 테크나 거대 회사를 위한 것"* 이라는 패배감도 깔린다.

그래도 대안은 더 나쁘다. 각 팀이 자기 컴포넌트, 색상, 패턴을 따로 만들면 분열된 경험이 되고, 결국 중앙 시스템으로 수렴해야 하는 전사적 마이그레이션이 닥친다.

### § 1.6 점진적 접근 — UI 라이브러리부터

저자는 이렇게 제안한다. *"처음부터 본격적인 디자인 시스템을 시작하지 마세요."*

멋진 웹사이트와 *"이 사이트의 규칙을 따라주세요"* 식 접근은 오히려 채택을 해친다. 대신 UI 라이브러리를 확장한다. 그게 결국 디자인 시스템의 한 조각이 된다.

이 방식이 좋은 이유는 현재 작업 방식과 어긋나지 않기 때문이다. 개발자는 매일 UI 라이브러리의 버튼을 가져다 쓴다. 그 버튼이 간격, 색상, 타이포그래피, 접근성에 대한 공유 결정을 반영하고 있으면, 디자인 시스템을 따로 의식하지 않아도 자동으로 일관성이 따라온다.

실제 현실의 흐름은 깨끗한 UI 라이브러리를 미리 만드는 일이 드물다는 것. 보통 기능을 먼저 만들고, 거기서 재사용 뷰를 추출하는 식으로 굴러간다.

> **Note** — UI 라이브러리 업데이트만으로 디자인 시스템을 지원했다고 하기엔 부족하다. 모든 기능이 마이그레이션됐다는 뜻이 아니기 때문. 그래도 훌륭한 출발점이다.

### § 1.7 현재 UI 라이브러리의 부족함 — 5가지 UI 프리미티브

지금 우리 UI 라이브러리는 작동은 하지만 깨지기 쉽다. 컬러는 하드코딩된 헥스로 흩뿌려져 있고(`#FF0044` vs `#FF2233`), 폰트 크기는 인라인으로 박혀 *"맞아 보이는"* 값(`16px` vs `15px`)이 들어가 있다. 아이콘은 중복되고 분열돼 있고, 간격은 케이스마다 다르다. 결과는 통일된 시스템이 아니라 컴포넌트 모음.

해결책으로 5가지 UI 프리미티브를 잡는다.

| 프리미티브 | 역할 |
|---|---|
| **색상 시스템** | 브랜드 정체성 반영, 컬러를 형식화된 언어로 중앙화 |
| **타이포그래피 스케일** | 폰트 패밀리·크기·스타일 표준화 |
| **통일된 아이콘 세트** | 중복 감소, 일관된 접근 |
| **간격 시스템** | 마진·패딩 스케일로 전반의 정렬 보장 |
| **미리 정의된 그림자** | 일관된 깊이 |

> UI 프리미티브는 뷰 프리미티브보다도 낮은 추상화 위에 산다 (Figure 1.5, 1.6).

---

### § 1.8 결론 — UI 라이브러리가 모든 출발점

플랫폼별 가이드라인(HIG/Material) 의 불일치를 지우는 게 목표는 아니다. 공유 요소는 표준화하면서 플랫폼 고유성은 존중한다. 목표는 완벽한 동일성이 아니라, 중요한 데서 일관성을 지키는 것.

많은 회사에서는 잘 유지되는 UI 라이브러리만으로도 충분하다. 풀 스케일 디자인 시스템 없이도 훌륭한 제품을 낼 수 있다. *"저노력" 디자인 시스템*, 그러니까 UI 컴포넌트를 중앙화하고 가벼운 문서를 붙이는 정도가 흔히 스위트 스폿이다. 풀 스케일 시스템이든 UI 라이브러리든, 출발점은 똑같다. 건강한 UI 라이브러리.

## Scale-02 · UI Library Fundamentals I (실용 챕터: 타이포그래피 + 컬러)

> *"UI 라이브러리는 컴파일되는 스타일 가이드입니다."*
> *"많은 팀에게는 잘 짜인 UI 라이브러리만으로도 조화롭게 일하기에 충분하다."*

추상화의 가장 낮은 수준. 책에서도 가장 낮은 수준이다. 코드 자체보다 길에서 내리는 결정이 더 중요하다.

---

### § 2.1 Typography — 프리미티브에서 시멘틱으로

타이포그래피는 두 층으로 잡는다.

| 층 | 이름 | 예 |
|---|---|---|
| 저수준 | **폰트 프리미티브** | font family, font size, font weight, line height, letter spacing |
| 고수준 | **시멘틱 텍스트 스타일** | `body`, `h1`, `h2`, `h3`, `caption`, `button` |

프리미티브를 직접 쓰면 안 되는 이유는 단순하다. *"Proxima-Nova"* 오타 하나로 깨진다. 누군가 20 대신 22, `.headline` 대신 `.title` 을 쓰면 바로 불일치가 생긴다. 그리고 모든 텍스트마다 모든 속성을 수동으로 설정해야 한다.

시멘틱이 주는 건 의도 중심의 사고다. 디테일이 아니라 의미에 집중하게 된다. 폰트 크기를 바꿔야 하면 한 곳만 고치면 끝. 디자이너가 정의를 주도하고, 개발자 혼자 결정하는 일이 없어진다.

구현 방식은 트레이드오프다.

| 접근 | 안전성 | 자유도 | 유지비 |
|------|------|------|------|
| `Text` 확장 (`text.companyFont(.h1)`) | 낮음 | 높음 | 낮음 |
| 풀 래퍼 (`CompanyText`) | 높음 | 낮음 | **높음** (네이티브 모디파이어 손실, 새 API 공짜로 못 받음) |
| **`TextStyle` 만 노출** (Compose 표준) | 중간 | 높음 | **낮음** (권장) |

> *구현은 디테일이다.* 핵심은 프리미티브와 시멘틱을 형식화하고, 작동하는 언어를 잡는 것.

**escape hatch.** 모든 변형을 시멘틱 스타일로 만들면 비대해진다(bloat). 시멘틱 + 네이티브 모디파이어 조합(예: `.companyFont(.body).bold()`)으로 균형을 잡는다.

---

### § 2.2 Colors — 같은 패턴

컬러 프리미티브는 디자이너에게서 받는 컬러 팔레트(스와치)다. 기능을 초월하는 공유 자원이고, `Colors.orangeColor = UIColor(hex: "#FFA500")` 처럼 고정·하드코딩된 표기로 잡힌다. 다크 모드, 테마, 접근성에 대해서는 아무것도 모른다.

프리미티브를 직접 쓰면 어떻게 되는가. 브랜딩이 주황에서 갈색으로 바뀌면 전 코드베이스를 find-and-replace 해야 한다. 커스텀 테마는 지원이 안 된다. 다크 모드도 안 된다. 그리고 컬러에 의미가 없다. `orangeColor` 는 왜 주황인지 말해주지 않는다.

해법은 시멘틱 컬러로 의미를 부여하는 것이다.

| 시멘틱 | 라이트 (프리미티브) | 다크 (프리미티브) |
|---|---|---|
| `warningColor` | `orangeColor` | `purpleColor` |
| `primaryBackgroundColor` | `whiteColor` | `lightGreyColor` |
| `accentColor` | `purpleColor` | `lightPurpleColor` |

*"중첩 상태의 양자 컬러"* 라는 표현이 잘 맞는다. 보기 전까지 실제 컬러를 모른다. 시멘틱 컬러는 그루핑이 아니라 동적으로 하나로 해소된다(이게 텍스트 스타일과 다른 점이다). 정리하면 이점이 셋이다. 의미가 붙고, swap 이 쉬워지고, 동적 컬러를 지원한다.

SwiftUI Asset 카탈로그도 비슷한 일을 하지만, 로컬 팔레트만 가능하고 프리미티브와 시멘틱을 구분하지 않는다. 작은 앱엔 충분해도 큰 앱에선 부족하다.

---

### § 2.3 컬러 추상화 분리 — Palette vs Colors

`Colors` struct 가 프리미티브와 시멘틱을 같이 담고 있으면 자동완성으로 실수가 난다. 시멘틱 컬러를 쓰려다 `Colors.orangeColor` 를 그대로 집어버린다.

명시적으로 분리한다.

```swift
struct Palette {
    static let orangeColor = UIColor(hex: "#FFA500")  // 프리미티브
    static let purpleColor = UIColor(hex: "#8A2BE2")
}

struct Colors {
    static let warningColor = UIColor { traits in     // 시멘틱
        traits.userInterfaceStyle == .dark ? Palette.purpleColor : Palette.orangeColor
    }
}
```

Palette 를 직접 만지는 건 시멘틱 컬러를 추가하거나 바꿀 때만이다.

**escape hatch.** 특정 기능에 전역 컬러를 추가하면 부담이 크다(브랜딩 승인, 멀티플랫폼). 이걸 피하려고 로컬 시멘틱 컬러를 정의할 수 있다. 예를 들어 기능 X 의 다크 모드 테두리만 보라색으로.

---

### § 2.4 Theming — 후드 아래의 swap

핵심 트릭은 이렇다. `Colors.warningColor` 의 콜 사이트는 변하지 않는다. 책임을 `Theme protocol` 로 위임한다.

```swift
protocol Theme {
    var warningColor: UIColor { get }
    var backgroundColor: UIColor { get }
}

struct DefaultTheme: Theme { /* ... */ }
struct CustomTheme: Theme  { /* ... */ }

struct Colors {
    static var currentTheme: Theme = DefaultTheme()
    static var warningColor: UIColor { currentTheme.warningColor }
}

// 런타임 전환
Colors.currentTheme = CustomTheme()
```

기능 엔지니어를 위한 코드 마이그레이션이 없으니 PM 이 행복하다. 고대비 테마나 접근성에도 그대로 쓸 수 있다. 한 가지 주의. 위 예시는 라이트/다크 동적 전환은 지원하지만, 런타임 테마 전환은 뷰 재빌드가 필요하다(앱 재시작 등).

이건 수천 가지 솔루션 중 하나일 뿐이다. 진짜 중요한 건, 시멘틱 컬러를 쓰는 순간 무한한 설정의 문이 열린다는 점.

---

### § 2.5 세밀한 컬러 제어 (Granular)

팔레트 변형은 두 갈래다.

```
Palette.red.lightest, Palette.red.lighter, Palette.red.base,
Palette.red.dark, Palette.red.darker, Palette.red.darkest
```

이름 기반은 의미는 살지만 확장이 어색해진다(`darkestest` 같은 이상한 이름이 나온다). 숫자 기반(`Palette.red.200`, `Palette.red.600`)은 의미가 약한 대신 미래 유연성이 좋다.

시멘틱 컬러도 그루핑해 둘 수 있다.

```
Colors.background.primary
Colors.background.secondary
Colors.background.tertiary
Colors.background.quaternary
```

더 인체공학적이다. 선투자가 들지만 자동완성이 깔끔해진다.

폰트와 컬러 모두 2층 구조로 분리한 결과, 더 우아한 코드와 더 쉬운 확장을 얻는다. 현실의 흐름은 처음부터 깨끗한 UI 시스템을 만드는 일이 드물다는 것. 보통 소급해서 기초 UI 를 추가한다. 다음 챕터에서 같은 패턴을 그림자, 아이콘, 간격에 적용하고, 그 다음에 디자인 시스템으로 연결된다.

## 나누고 싶은 이야기

### 1. 추가로 학습한 것 — Jetpack Compose 매핑

책 예제는 SwiftUI 기반이라 Compose 등가 구현을 따로 정리했다.

타이포그래피 쪽은 이렇다. SwiftUI 의 `Text.companyFont(.h1)` (Modifier 체인) 은 Compose 에서 `Text(text=..., style=CompanyTypography.h1)` (파라미터 주입) 으로 바뀐다. `MaterialTheme.typography` 슬롯에 회사 토큰을 덮어쓰면 모든 자식 컴포저블이 자동으로 적용을 받는다. 채택률이 올라간다.

`CompanyText` 같은 풀 래퍼는 함정이 있다. `Text()` 는 17개 파라미터를 갖는다. 래퍼는 그걸 수동으로 따라가야 한다. `maxLines`, `overflow`, `textAlign` 이 한 달, 두 달, 세 달 간격으로 추가되는 파라미터 추가 지옥이 펼쳐진다.

결론은, Compose 커뮤니티 표준은 `TextStyle` 만 시멘틱화하고 `Text` 는 그대로 쓰는 절충안이다. 시멘틱(스타일) vs 동작(레이아웃 파라미터) 의 책임을 나눈 것. 이게 §2.1 트레이드오프 표에서 결국 세 번째 줄(TextStyle 만 노출)이 이기는 이유다.

### 2. *"중간 규모 회사에선 UI 라이브러리만으로도 충분하다"* — 최근의 고민

§ 1.8 과 § 2.1 도입부에서 저자가 반복해서 짚는 부분이 최근 고민과 정확히 맞물렸다.

> *"많은 회사에서, 잘 유지되는 UI 라이브러리는 대부분의 문제를 해결하고 훌륭한 제품을 출시하기에 충분합니다."*
>
> *"많은 팀에게는 잘 짜인 UI 라이브러리만으로도 조화롭게 일하기에 충분할 수 있다고 주장하고 싶습니다."*

이게 왜 위로가 됐는가. "디자인 시스템 도입" 이라는 단어를 들으면 반짝이는 웹사이트, 거버넌스 회의, 토큰 파이프라인, 멀티플랫폼 동기화 같은 벅찬 그림이 먼저 떠오른다. 그런데 저자는 *"처음부터 본격적인 디자인 시스템을 시작하지 마세요"* 라고 못박는다. 채택을 해친다는 이유로. 대신 현재 작업 방식과 어긋나지 않는 점진적 확장을 제안한다. UI 라이브러리에 프리미티브와 시멘틱 두 층을 단단히 세우는 것. 그게 스위트 스폿이라는 것.

실무에 가져온다면 이렇게 된다. *"우리도 디자인 시스템 만들어야 하는 거 아닌가?"* 라는 질문에 대한 답은, 먼저 UI 라이브러리가 5가지 프리미티브를 단단히 갖고 있는지 점검하는 일이다. 토큰, 문서, 거버넌스는 그 다음. 또는 영영 안 가도 된다. 마이그레이션 압박 없이 점진적으로 추가할 수 있다. 새 기능에서 시멘틱 컬러를 쓰고, 레거시는 기회 있을 때 정리하면 된다.

### 3. *디자인 시스템 ≠ 버튼·폰트·컬러 모음집*

오늘의 가장 큰 멘탈 모델 업데이트다.

기존 머릿속의 디자인 시스템은 *"재사용 가능한 버튼, 정해진 폰트 스케일, 약속된 컬러 팔레트 모음"* 정도였다. 책이 알려준 진짜 그림은 다르다 (§ 1.5 Figure 1.2).

```
디자인 시스템
├── UI 라이브러리 (다중 구현 — iOS, Android, Web)
├── 토큰 (디자인 결정을 명명된 값으로 공식화)
├── 문서 (사용 가이드, 원칙, 규칙)
├── 디자인 자산 (아이콘, 컴포넌트, 템플릿, 로고, 일러스트레이션)
├── 거버넌스 (시스템 변경·승인 프로세스, 기여 규칙)
└── 툴링 & 자동화 (Figma ↔ 코드 동기화, 토큰 빌드 파이프라인 등)
```

여기서 잡힌 깨달음 몇 가지. UI 라이브러리는 디자인 시스템의 부분집합이다. 단독으로 존재할 수도 있지만, 디자인 시스템 안에서 보면 한 조각일 뿐이다. 진실의 원천 위치도 다르다. UI 라이브러리는 *코드*, 디자인 시스템은 *문서*. 그래서 소유권도 달라진다 (개발자 vs 팀들).

디자인 시스템의 본질은 "사용의 why·how" 다. 단순히 재사용 가능한 버튼이 있다가 아니라, primary vs secondary 를 언제 쓸지, 접근성을 어떻게 보장할지, 어떤 시각 위계를 유지할지가 핵심이다.

거버넌스는 빼먹기 쉬운데, 디자인 시스템이 시간이 지나도 망가지지 않게 지키는 규칙과 프로세스가 시스템의 핵심 자산이다.

그리고 대화의 결이 바뀐다. *"이 컴포넌트는 그것을 염두에 두고 만들지 않았어요"* 에서 *"일관성을 유지하면서 새 요구를 만족시키도록 어떻게 진화시킬까?"* 로.

그래서 결론은 이렇다. 디자인 시스템을 만든다는 건 버튼·폰트·컬러를 정리한다는 뜻이 아니다. 제품 전체의 UI 의사결정을 공식화하고, 그걸 유지하고 진화시킬 팀·시스템 양쪽의 절차를 만든다는 뜻이다. 건강한 UI 라이브러리가 출발점이지만, 그게 디자인 시스템 자체는 아니다.

그래서 위의 #2(중간 규모 회사에선 UI 라이브러리로 충분)와 #3 이 동전의 양면이다. 디자인 시스템 전체를 짓는 벅찬 과제 대신, UI 라이브러리라는 그 핵심 조각을 단단하게 만드는 것. 대부분의 팀이 가야 할 길은 거기다.
