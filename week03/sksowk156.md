# 5장

# 토큰으로 UI 결정을 표준화하기

디자인 시스템을 이야기할 때 자주 등장하는 개념이 **토큰**이다.

토큰은 색상, 간격, 타이포그래피, 아이콘, 그림자 같은 UI 결정을 이름 붙은 값으로 관리하는 방식이다. 이를 통해 UI 결정을 재사용하고, 디자인과 코드 사이에서 동기화하며, 시스템이 커져도 일관성을 유지할 수 있다.

디자인 시스템의 목적은 단순히 화면을 통일하는 것이 아니다. 디자이너와 개발자가 UI를 같은 기준으로 이해하고, 같은 방식으로 협업할 수 있게 만드는 것이다.

---

## UI 요소를 토큰으로 묶는 이유

UI를 만들다 보면 같은 속성을 여러 이름으로 부르게 된다.

예를 들어 색상 하나도 상황에 따라 다음처럼 다르게 표현될 수 있다.

- 컬러 프리미티브
- 시맨틱 컬러
- 테두리 색상
- 경고 색상
- 실제 색상 값

이런 용어가 계속 늘어나면 협업 비용이 커진다. 디자이너와 개발자가 같은 값을 두고도 서로 다른 개념을 이야기할 수 있기 때문이다.

그래서 색상, 간격, 텍스트 스타일, 아이콘 같은 UI 결정을 하나의 공통 개념으로 묶는다. 그 개념이 **토큰**이다.

---

## 토큰이란 무엇인가?

토큰은 구조화된 형식으로 저장된 재사용 가능한 디자인 결정이다.

다르게 말하면, 앱이나 웹사이트 전체에서 디자인을 일관되게 유지하기 위한 **이름 붙은 값**이다.

| 토큰 이름 | 값 |
| --- | --- |
| Primary Color | `#0057B8` |
| Alert Color | `#FF5733` |
| Spacing Small | `8px` |
| Spacing Medium | `16px` |

토큰을 사용하면 화면마다 값을 직접 지정하지 않아도 된다. 예를 들어 여러 화면에서 `#0057B8`을 직접 쓰는 대신 `Primary Color`라는 이름을 사용한다.

값이 바뀌어야 할 때도 토큰 정의만 수정하면 된다. UI 코드는 실제 값보다 이름에 의존하게 된다.

---

## 이미 토큰을 사용하고 있었을 수 있다

코드에서 `Color.primaryBackground`나 `Shadow.large`처럼 시맨틱한 이름을 가진 UI 요소를 정의했다면, 이미 토큰과 비슷한 구조를 사용하고 있는 것이다.

중요한 점은 실제 값을 직접 쓰지 않고, 값의 의미를 이름으로 표현하는 것이다.

예를 들어 다음과 같은 값이 있다고 하자.

```kotlin
object AppColors {
    val PrimaryBackground = Color(0xFFFFFFFF)
    val AlertBackground = Color(0xFFFF5733)
}

object AppSpacing {
    val Small = 8.dp
    val Medium = 16.dp
}
```

여기서 `PrimaryBackground`, `AlertBackground`, `Small`, `Medium`은 단순한 값이 아니라 UI 결정을 표현하는 이름이다.

---

## 프리미티브 토큰과 시맨틱 토큰

토큰은 크게 두 종류로 나눌 수 있다.

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| 프리미티브 토큰 | 의미가 붙지 않은 원시 값 | `Palette.purple` |
| 시맨틱 토큰 | 사용 의도가 붙은 값 | `Icons.delete` |

`Palette.purple`은 보라색이라는 값만 나타낸다. 어디에 쓰이는지는 알 수 없다. 따라서 프리미티브 토큰이다.

반면 `Icons.delete`는 아이콘이 어떻게 생겼는지를 말하지 않는다. “삭제”라는 행동을 나타낸다.

실제 아이콘이 휴지통 모양에서 X 모양으로 바뀌어도 `Icons.delete`를 사용하는 코드는 바뀌지 않는다. 토큰 값만 바꾸면 된다.

시맨틱 토큰의 장점은 값이 아니라 의도를 기준으로 코드를 작성할 수 있다는 점이다.

---

## 토큰을 쓴다고 디자인 시스템은 아니다

토큰을 사용한다고 해서 곧바로 디자인 시스템을 사용하는 것은 아니다.

UI 라이브러리는 보통 개발팀 안에서 관리된다. 개발자는 색상, 간격, 컴포넌트 등을 코드로 정리하고 재사용한다.

반면 디자인 시스템은 더 넓은 협업 체계다. 디자인, 개발, 브랜딩, Android, iOS, Web 같은 여러 팀이 함께 참여한다.

핵심 차이는 **공동 소유권**이다.

디자인 시스템에서는 디자이너와 개발자가 토큰을 함께 소유한다. 디자이너는 토큰의 값뿐만 아니라 이름, 분류, 그룹화에도 참여한다. 개발자는 코드 구조와 유지보수성을 고려해 토큰을 구현한다.

즉, 토큰은 도구이고, 디자인 시스템은 그 토큰을 함께 정의하고 관리하는 방식까지 포함한다.

---

## 토큰 이름은 의도를 설명해야 한다

토큰 이름은 팀의 공통 언어다.

좋은 토큰 이름은 값의 외형보다 사용 의도를 설명해야 한다.

예를 들어 `orangeColor`는 색상의 외형을 설명한다. 반면 `warningBackground`는 그 색상이 어떤 상황에 쓰이는지 설명한다.

토큰 이름을 정할 때는 다음 질문을 기준으로 삼을 수 있다.

> 이 이름은 무엇을 의미하는가?
>
>
> 아니면 단지 어떻게 보이는지만 설명하는가?
>

시맨틱 토큰은 가능하면 의미를 담아야 한다. 그래야 나중에 실제 값이 바뀌어도 이름을 유지할 수 있다.

예를 들어 경고 배경색이 주황색에서 빨간색으로 바뀌어도 `warningBackground`라는 이름은 그대로 사용할 수 있다. 하지만 `orangeBackground`라고 이름을 지었다면 값이 바뀌는 순간 이름도 어색해진다.

---

## 토큰은 어디에 있어야 할까?

토큰은 디자이너, 개발자, 다른 팀원이 함께 접근하고 업데이트할 수 있는 위치에 있어야 한다.

토큰은 다음 위치에 있을 수 있다.

- 디자인 도구
- 코드베이스
- 문서
- 공유 저장소

중요한 것은 토큰의 **정의**와 **구현**을 구분하는 것이다.

코드 안에 있는 토큰은 실제 앱에서 사용되는 구현에 가깝다. 반면 토큰의 정의는 보통 디자인에서 시작된다.

따라서 토큰이 코드 안에 있더라도 개발자만의 소유물로 두면 안 된다. 디자이너가 토큰에 영향을 주고 업데이트할 수 있는 워크플로우가 필요하다.

---

## 토큰 동기화의 문제

토큰을 운영할 때 중요한 문제는 디자인과 코드가 어긋나지 않게 유지하는 것이다.

디자이너가 색상이나 간격을 수정했는데 코드가 그대로라면 디자인과 실제 앱이 달라진다. 반대로 개발자가 코드에서 값을 임의로 바꾸면 디자인 파일과 구현이 달라진다.

작은 팀에서는 자동화 없이 수동 동기화 방식으로 시작할 수 있다.

대표적인 방식은 세 가지다.

1. 디자인을 기준으로 개발자가 코드에 반영한다.
2. 코드를 기준으로 디자이너가 토큰을 수정한다.
3. 핸드오프 문서로 변경 사항을 전달한다.

각 방식은 장단점이 있다.

---

## 방식 1. 디자인을 기준으로 개발자가 동기화하기

첫 번째 방식은 디자인 파일을 단일 기준점으로 삼는 것이다.

디자이너가 디자인을 업데이트하면, 개발자가 디자인 파일에서 변경 사항을 확인하고 코드의 토큰을 수정한다.

```
디자인 파일 변경
→ 개발자가 변경 사항 확인
→ 코드의 토큰 수정
→ 빌드 확인
```

이 방식에서는 디자인 파일이 기준이다. 코드와 디자인이 다르면 코드는 디자인을 따라가야 한다.

장점은 디자이너가 개발 환경에 들어오지 않아도 된다는 점이다. 디자이너는 디자인 도구에서 작업하고, 개발자는 그 결과를 코드로 옮긴다.

문제는 개발자가 디자인 파일에서 정확한 변경 사항을 찾아야 한다는 점이다. 디자인 파일이 정리되어 있지 않으면 어떤 색상, 어떤 레이어, 어떤 스타일이 기준인지 알기 어렵다.

이 방식이 작동하려면 디자인 파일 안에 개발자가 쉽게 찾을 수 있는 스타일 목록이 있어야 한다.

예를 들어 다음이 정리되어 있어야 한다.

- 색상 팔레트
- 타이포그래피 목록
- 간격 기준
- 변경 전/후 기록

디자이너가 변경 기록을 관리해주면 개발자는 토큰을 더 쉽게 코드에 반영할 수 있다.

---

## 방식 2. 코드를 기준으로 디자이너가 동기화하기

두 번째 방식은 코드베이스를 단일 기준점으로 삼는 것이다.

디자이너가 디자인을 수정한 뒤, 코드 안의 토큰 정의 파일도 직접 수정한다. 그리고 그 변경 사항을 pull request로 올린다.

```
디자인 변경
→ 디자이너가 코드의 토큰 파일 수정
→ pull request 생성
→ 개발자가 검토
→ 머지
```

이 방식의 전제는 코드가 토큰을 더 중앙화해서 관리하고 있다는 것이다. 코드에서는 보통 관련 값이 한곳에 모여 있고, 나머지 코드가 그 정의를 참조한다.

장점은 별도의 동기화 도구를 만들지 않아도 된다는 점이다. 디자이너가 어떤 토큰을 바꿔야 하는지 알고 있다면 직접 변경을 제안할 수 있다.

하지만 이 방식은 디자이너가 개발 환경에 들어와야 한다. 단일 플랫폼이나 작은 팀에서는 가능할 수 있지만, 플랫폼이 늘어나면 부담이 커질 수 있다.

또한 토큰 값을 바꾸는 것과 토큰을 삭제하거나 이름을 바꾸는 것은 다르다.

값 변경은 비교적 단순하다.

하지만 토큰을 삭제하거나 이름을 바꾸면 기존 코드가 깨질 수 있다. 이미 여러 화면에서 그 토큰을 사용하고 있을 수 있기 때문이다.

이런 경우에는 바로 삭제하기보다 deprecated 처리 후 점진적으로 대체하는 편이 안전하다.

토큰은 여러 화면이 공유하는 값이다. 그래서 이름 변경과 삭제는 영향 범위를 신중하게 봐야 한다.

---

## 디자이너와 개발자 사이의 책임 균형

토큰 동기화는 디자이너와 개발자의 협업을 필요로 한다. 하지만 책임이 한쪽으로 치우치면 문제가 생긴다.

디자이너가 직접 코드를 수정하는 방식은 디자이너에게 부담을 줄 수 있다. 한 명의 디자이너가 시각 디자인, 토큰 값 수정, pull request 작성, 리뷰 대응까지 모두 맡게 될 수 있다.

반대로 개발자가 디자인 파일에서 모든 토큰 변경을 추출하는 방식도 한계가 있다. 이 경우 디자이너는 자신의 변경 사항이 코드에서 어떻게 해석되고, 어떤 이름으로 바뀌고, 어떤 우선순위로 반영되는지 보기 어렵다.

따라서 디자이너는 의도와 변경 사항을 명확히 제공하고, 개발자는 코드 구조와 유지보수성을 고려해 반영하는 구조가 필요하다.

---

## 방식 3. 핸드오프 문서 사용하기

세 번째 방식은 핸드오프 문서를 사용하는 것이다.

앞선 두 방식은 디자이너나 개발자 중 한쪽이 상대방의 환경으로 들어가야 한다. 개발자가 디자인 파일을 깊게 확인하거나, 디자이너가 코드베이스에 들어가야 한다.

핸드오프 방식은 각자가 자기 도구 안에서 작업하면서 변경 사항을 공유하는 방식이다.

디자이너는 토큰 변경 사항을 “Before and After” 문서로 정리한다. 개발자는 이 문서를 보고 코드의 토큰을 업데이트한다.

| 토큰 | 변경 전 | 변경 후 |
| --- | --- | --- |
| `tertiaryButtonBackground` | `#4CAF50` | `#388E3C` |
| `spacing.medium` | `12` | `10` |

이 방식의 장점은 변경 사항을 공통 형식으로 전달할 수 있다는 점이다. 디자이너는 디자인 도구 안에 머물고, 개발자는 코드베이스 안에서 작업한다.

핸드오프 문서는 단일 기준점이 아니다. 변경 사항을 명확히 전달하기 위한 커뮤니케이션 도구다.

이 방식은 변경 사항이 많지 않고, 자주 발생하지 않을 때 잘 작동한다.

---

## 토큰 결정은 공동으로 이루어진다

핸드오프 문서가 있다고 해서 디자이너가 토큰을 일방적으로 결정하는 것은 아니다. 토큰 결정은 여전히 디자이너와 개발자가 함께 맞춰야 한다.

디자이너는 `modalOverlayBackground`나 `navigationBarShadow`처럼 새로운 토큰을 제안할 수 있다. 하지만 개발자는 그 토큰이 기존 시스템과 어떻게 연결되는지 확인해야 한다.

예를 들어 이미 비슷한 역할을 하는 토큰이 있을 수 있다. 또는 새 토큰이 특정 플랫폼의 기본 동작과 맞지 않아 구현을 복잡하게 만들 수도 있다.

따라서 핸드오프 문서는 최종 명세라기보다 논의를 시작하기 위한 제안에 가깝다. 디자이너는 토큰의 의도와 사용 맥락을 전달하고, 개발자는 기존 시스템과 구현 관점에서 검토한다.

이 과정에서 개발자는 단순히 전달받은 값을 코드에 옮기는 사람이 아니다. 디자인 시스템이 일관성을 유지하고, 장기적으로 관리 가능한 구조가 되도록 함께 기여한다.

---

## 정리

토큰은 UI 결정을 이름과 구조로 관리하는 방식이다. 색상, 간격, 타이포그래피, 아이콘 같은 값을 직접 사용하는 대신, 의미 있는 이름으로 참조하게 만든다.

토큰에는 프리미티브 토큰과 시맨틱 토큰이 있다. 프리미티브 토큰은 원시 값이고, 시맨틱 토큰은 사용 의도를 담은 값이다.

토큰을 쓴다고 곧바로 디자인 시스템이 되는 것은 아니다. 디자인 시스템은 디자이너와 개발자가 토큰을 함께 정의하고, 관리하고, 동기화하는 협업 체계다.

토큰 운영에서 중요한 것은 다음이다.

- 토큰 이름은 값보다 의도를 설명해야 한다.
- 토큰의 정의와 구현을 구분해야 한다.
- 디자인과 코드가 어긋나지 않도록 동기화 방식이 필요하다.
- 책임이 디자이너나 개발자 한쪽으로 치우치면 안 된다.
- 변경이 적을 때는 수동 핸드오프도 가능하다.
- 변경이 많아지고 플랫폼이 늘어나면 중앙화와 자동화가 필요하다.

디자인 시스템의 목표는 도구를 세팅하는 것이 아니다. 여러 팀이 같은 기준으로 UI를 이해하고, 변경 사항을 일관되게 반영할 수 있는 워크플로우를 만드는 것이다.

---

## 후기

이 장에서 제시한 방식 중 “디자이너가 코드의 토큰을 직접 수정한다”는 접근은 작은 팀이나 단일 플랫폼에서는 가능할 수 있다. 하지만 Android, iOS, Web처럼 여러 플랫폼을 함께 운영하는 환경에서는 부담이 커진다.

핵심 문제는 디자이너가 코드를 작성할 수 있느냐가 아니다. 플랫폼별 토큰 파일을 모두 찾아 수정해야 하고, 그 과정에서 실수 가능성과 작업 시간이 늘어난다는 점이다. 토큰 코드가 단순하더라도 플랫폼마다 구조와 규칙이 다르면, 디자이너가 모든 구현 세부 사항을 안정적으로 다루기는 어렵다.

또한 “디자이너가 수정한 코드가 컴파일되지 않아도 된다”는 관점은 조심해서 받아들일 필요가 있다. 개발자가 나중에 문법 오류를 고칠 수는 있다. 하지만 변경 의도가 명확하지 않다면 개발자마다 다르게 해석할 수 있다. 결국 중요한 것은 코드가 완성되어 있느냐가 아니라, 어떤 토큰을 어떤 의도로 바꾸려는지가 명확하게 전달되는 것이다.

그래서 디자이너가 직접 코드를 완성하는 방식보다는, 변경 의도를 명확히 전달하고 개발자가 각 플랫폼 구조에 맞게 반영하는 방식이 더 안정적이라고 생각한다.

이 장에서는 “토큰 결정은 여전히 디자이너와 개발자가 함께 내린다”고 설명한다. 이어서 나온 예시는 새 토큰을 만들지, 기존 토큰을 재사용할지, 더 적합한 토큰 이름을 제안할지 판단하는 상황에 가깝다.

하지만 나는 이런 결정을 개발자가 함께 내려야 할 영역으로 보기는 어렵다고 생각한다. 개발자가 함께 참여해야 하는 부분은 개별 색상 값이나 특정 토큰의 사용 여부를 판단하는 일이 아니라, 토큰을 어떤 기준으로 만들고 관리할지 정하는 일에 가깝다고 생각한다.

즉, 디자이너와 개발자가 함께 합의해야 할 것은 토큰의 생성 기준, 분류 방식, 네이밍 규칙이다. 그 기준이 정해져 있다면 같은 색상 값을 사용하더라도 그 의도가 다르다고 판단할 수 있고 별도의 시맨틱 토큰으로 분리할 수 있다고 생각한다.

결국 개발자의 역할은 디자인 의사결정을 대신하는 것이 아니라, 토큰 체계가 코드에서도 일관되고 유지보수 가능한 구조가 되도록 기준을 함께 설계하는 것이라고 본다.

# 6장

# 확장 가능한 디자인 시스템: 플랫폼과 팀을 위한 토큰 워크플로우

5장에서는 단일 플랫폼에서 토큰을 어떻게 정의하고 동기화할지 다뤘다. 하지만 실제 제품은 Android 하나만으로 끝나지 않는 경우가 많다. iOS, Web, 여러 앱, 여러 팀이 함께 움직이기 시작하면 수동으로 토큰을 맞추는 방식은 금방 한계에 부딪힌다.

6장의 핵심은 토큰 워크플로우를 여러 플랫폼과 팀으로 확장하는 방법이다. 처음에는 수동 핸드오프에서 시작하지만, 점점 중앙 토큰 저장소, 문서화, 검증, 코드 생성, CI 자동화로 발전한다.

---

## 단일 플랫폼에서는 수동 핸드오프도 가능하다

디자인 팀과 개발 팀 사이에서 가끔 발생하는 토큰 변경은 수동 핸드오프로도 처리할 수 있다.

이때 중요한 전제는 디자인 파일이 단일 기준점이라는 점이다. 코드와 디자인이 어긋났다면, 코드의 UI가 다시 디자인을 따라가야 한다.

```
디자인 파일 변경
→ 디자이너가 변경 사항 전달
→ 개발자가 코드 토큰 수정
→ 빌드 및 검수
```

이 방식은 변경이 적고 플랫폼이 하나일 때는 단순하다. 하지만 플랫폼이 늘어나면 문제가 생긴다.

---

## 멀티 플랫폼에서는 핸드오프 비용이 커진다

Android와 iOS를 함께 운영한다면 디자이너는 같은 변경 사항을 여러 팀에 전달해야 한다.

변경이 자주 발생하지 않는다면 괜찮을 수 있다. 하지만 팀과 플랫폼이 늘어나면 디자이너가 여러 팀의 반영 여부를 계속 확인해야 한다.

예를 들어 다음과 같은 일이 생긴다.

- Android는 반영했지만 iOS는 아직 반영하지 않음
- 어떤 릴리즈에 새 토큰이 포함되는지 알기 어려움
- 변경 문서가 Slack, Confluence, Google Docs 등에 흩어짐
- 플랫폼마다 토큰 이름이나 값이 조금씩 달라짐

이런 상황에서는 토큰이 점점 어긋난다. 이를 **토큰 드리프트(token drift)**라고 볼 수 있다.

토큰 드리프트는 같은 디자인 의도가 플랫폼마다 다른 이름이나 다른 값으로 관리되면서 일관성이 깨지는 현상이다.

---

## 중앙 토큰 저장소가 필요한 이유

시스템이 커지면 토큰을 한곳에서 관리해야 한다. 이를 위해 **토큰 저장소**를 둔다.

토큰 저장소는 토큰을 구조화해서 저장하고 버전 관리하는 공간이다. 보통 Git 저장소를 사용하며, 토큰은 JSON 같은 플랫폼 독립적인 형식으로 저장한다.

토큰 저장소는 디자인 팀과 개발 팀 사이의 단일 기준점이 된다.

토큰 저장소를 사용하면 다음 이점이 있다.

- 변경 이력을 추적할 수 있다.
- 토큰 이름과 형식을 검증할 수 있다.
- Android, iOS, Web용 결과물을 생성할 수 있다.
- 디자인과 개발이 같은 토큰 기준을 사용할 수 있다.
- 여러 제품과 플랫폼으로 디자인 시스템을 확장하기 쉬워진다.

핵심은 토큰을 디자인 파일이나 각 플랫폼 코드 안에만 두지 않고, 공통 기준점으로 분리하는 것이다.

---

## 토큰은 JSON으로 관리할 수 있다

여러 플랫폼이 토큰을 공유하려면 특정 플랫폼에 종속되지 않는 형식이 필요하다. 가장 흔한 선택은 JSON이다.

JSON은 사람이 읽기 쉽고, 디자인 도구나 자동화 스크립트에서 다루기 쉽다. 또한 Android, iOS, Web 중 어느 한쪽의 문법에 묶이지 않는다.

예를 들어 토큰 JSON은 다음처럼 구성할 수 있다.

```json
{
  "color": {
    "background": {
      "primary": "#007AFF",
      "secondary": "#F2F2F7"
    },
    "text": {
      "primary": "#000000",
      "secondary": "#666666"
    }
  },
  "spacing": {
    "small": "8",
    "medium": "16",
    "large": "24"
  },
  "font": {
    "header": {
      "size": "20",
      "weight": "regular"
    },
    "body": {
      "size": "16",
      "weight": "light"
    }
  }
}
```

이 구조는 Android의 `dp`, iOS의 `pt`, Web의 `px`를 직접 가정하지 않는다. 먼저 공통 토큰을 정의하고, 각 플랫폼에서 필요한 형태로 변환한다.

예를 들어 같은 JSON을 기반으로 다음 결과물을 만들 수 있다.

```
JSON
→ Android XML 또는 Kotlin
→ iOS Swift
→ Web CSS variables
```

---

## 디자이너는 토큰을 export도 하고 import도 해야 한다

토큰 저장소가 단일 기준점이라면 디자이너는 디자인 도구도 이 저장소와 맞춰야 한다.

디자이너가 디자인 도구에서 토큰을 수정했다면, 그 토큰을 JSON으로 export해서 저장소에 반영해야 한다.

반대로 저장소의 토큰이 변경되었다면, 디자이너는 그 토큰을 다시 디자인 도구로 import해야 한다. 그렇지 않으면 코드와 저장소는 최신인데 디자인 파일은 이전 값을 계속 사용할 수 있다.

즉, 토큰 저장소를 기준으로 디자인 도구와 플랫폼 코드가 모두 동기화되어야 한다.

이 구조가 있어야 디자이너와 개발자가 같은 토큰을 기준으로 작업할 수 있다.

---

## Git을 사용하면 변경 이력을 추적할 수 있다

토큰이 디자인 도구 안에만 있으면 변경 이력이 잘 보이지 않는다. UI가 달라졌다는 것은 알 수 있지만, 어떤 토큰이 왜 바뀌었는지 추적하기 어렵다.

토큰을 Git 저장소에서 관리하면 다음이 가능해진다.

- 어떤 토큰이 변경되었는지 확인
- 변경 전후 diff 확인
- 변경 이유를 pull request에 기록
- 특정 버전으로 되돌리기
- 릴리즈 단위로 토큰 버전 관리

다만 Git에서 토큰 JSON을 되돌린다고 해서 디자인 도구의 화면이 자동으로 되돌아가는 것은 아니다. 롤백된 토큰을 다시 디자인 도구에 import하거나, 디자인 도구 자체의 버전 기록을 사용해야 한다.

---

## 토큰 aliasing

토큰은 실제 값을 직접 가질 수도 있고, 다른 토큰을 참조할 수도 있다. 이 방식을 **aliasing**이라고 한다.

Aliasing은 어떤 토큰이 실제 값이 아니라 다른 토큰을 참조하도록 만드는 방식이다.

예를 들어 다음 JSON을 보자.

```json
{
  "color": {
    "grey": {
      "light": { "value": "#cccccc" }
    },
    "background": {
      "value": "{color.grey.light}"
    }
  }
}
```

여기서 `color.grey.light`는 실제 색상 값이다.

`color.background`는 직접 값을 갖지 않고 `color.grey.light`를 참조한다.

이렇게 하면 원시 값과 시맨틱 토큰을 분리할 수 있다.

---

## 플랫폼 간 토큰 공유 전에 이름을 맞춰야 한다

중앙 토큰 저장소를 사용하려면 먼저 토큰 이름을 맞춰야 한다.

여기서 이름을 맞춘다는 것은 모든 플랫폼이 같은 값을 사용해야 한다는 뜻이 아니다. 같은 디자인 의도를 같은 이름으로 부르자는 뜻이다.

예를 들어 Android에서는 그룹 간격을 `groupSpacing`, iOS에서는 `sectionPadding`이라고 부를 수 있다. 각 플랫폼 안에서는 문제가 없을 수 있다. 하지만 디자인 팀이 “그룹 간격을 늘리자”고 말했을 때 어떤 토큰을 바꿔야 하는지 바로 알기 어렵다.

공통 토큰 이름이 없다면 다음 문제가 생긴다.

- 플랫폼마다 같은 의도를 다른 이름으로 관리한다.
- 디자인 변경을 여러 플랫폼에 전달하기 어렵다.
- 공통 JSON 파일을 만들기 어렵다.
- 변경 사항을 추적하기 어렵다.

따라서 플랫폼 간 공유가 필요한 토큰은 먼저 이름을 맞춰야 한다.

---

## 토큰 이름과 값은 분리해서 생각해야 한다

플랫폼이 같은 토큰 이름을 사용한다고 해서 반드시 같은 값을 사용해야 하는 것은 아니다.

토큰 이름은 실제 값이 아니라 디자인 의도를 나타낸다.

예를 들어 `Font.header`는 “주요 헤더 스타일”이라는 의도를 나타낸다. 하지만 실제 값은 플랫폼마다 다를 수 있다.

| 토큰 | iOS | Android |
| --- | --- | --- |
| `Font.header` | size 20, regular | size 18, bold |
| `Spacing.small` | 8pt | 10dp |

이름은 공유하되, 실제 값은 플랫폼별로 다르게 가져갈 수 있다. 이를 위해 override와 extension이 필요하다.

---

## 플랫폼별 override

모든 플랫폼의 값을 하나의 `tokens.json`에 넣을 수도 있다. 하지만 iOS와 Android 전용 값이 한 파일에 섞이면 관리가 어려워진다.

대신 파일을 분리하는 편이 낫다.

```
tokens/
  base.json      # 공유 기본값
  ios.json       # iOS 전용 override
  android.json   # Android 전용 override
```

`base.json`에는 공통 기본값을 둔다.

`android.json`과 `ios.json`에는 각 플랫폼에서 달라지는 값만 둔다.

이 구조를 사용하면 공유 토큰은 플랫폼 중립적으로 유지하고, 플랫폼별 차이는 별도 파일에서 관리할 수 있다.

---

## 플랫폼별 extension

Override는 공통 토큰의 값을 플랫폼별로 바꾸는 방식이다. 하지만 어떤 토큰은 특정 플랫폼에만 필요하다.

예를 들어 Android에는 elevation 개념이 자연스럽다.

반면 iOS에서는 같은 깊이감을 shadow나 다른 방식으로 표현할 수 있다.

이런 Android 전용 토큰을 공통 토큰에 억지로 넣으면 iOS도 필요 없는 토큰을 보게 된다. 반대로 iOS 전용 개념을 Android가 볼 필요도 없다.

이럴 때는 플랫폼별 extension을 사용한다.

```json
{
  "elevation": {
    "low": { "value": "2" },
    "medium": { "value": "8" }
  }
}
```

이 토큰은 `android.json`에만 있어도 된다.

정리하면 다음과 같다.

| 구분 | 설명 |
| --- | --- |
| base token | 여러 플랫폼이 공유하는 토큰 |
| override | 공유 토큰의 값을 플랫폼별로 덮어쓰기 |
| extension | 특정 플랫폼에만 필요한 토큰 추가 |

---

## 문서화가 필요한 이유

토큰이 JSON 저장소에 있다고 해서 모두가 토큰을 이해하는 것은 아니다.

문서의 역할은 토큰 정의에 맥락과 사용 가이드를 추가하는 것이다.

문서에는 다음 내용이 들어가면 좋다.

- 토큰의 목적
- 사용해야 하는 곳
- 사용하지 말아야 하는 곳
- 플랫폼별 참고 사항
- 시각적 예시
- 변경 전후 예시

예를 들어 다음처럼 작성할 수 있다.

```markdown
## color.background.primary

값:
- light mode: #FFFFFF
- dark mode: #000000

목적:
- 콘텐츠 영역의 기본 배경

사용처:
- 화면 배경
- 카드 배경

사용하지 말 것:
- 텍스트
- 테두리
- 버튼 배경

플랫폼 참고:
- Android에서는 surface 계열 색상과 구분해서 사용한다.
```

문서는 화려할 필요가 없다. 중요한 것은 찾기 쉽고 최신 상태여야 한다는 점이다.

```
design-tokens/
  tokens/
    base.json
    android.json
    ios.json
  TOKENS.md
```

---

## 문서는 누가 관리해야 할까?

문서는 디자이너와 개발자가 함께 관리하는 편이 좋다.

보통 디자이너가 토큰의 의도와 사용 방법을 먼저 정리한다. 개발자는 구현하면서 헷갈리는 이름, 플랫폼별 차이, 접근성 관련 내용을 추가할 수 있다.

처음부터 모든 토큰을 문서화할 필요는 없다. 자주 질문이 나오는 토큰, 오용이 잦은 토큰부터 정리하면 된다.

---

## 자동화는 언제 필요할까?

토큰 저장소가 생겼다고 바로 자동화를 도입할 필요는 없다.

먼저 확인해야 할 질문은 다음이다.

- 토큰 업데이트가 얼마나 자주 발생하는가?
- 즉시 동기화되지 않으면 문제가 되는가?
- 수동 복사 때문에 실수가 반복되는가?
- 여러 플랫폼이 같은 시점에 업데이트되어야 하는가?
- 토큰 변경을 조율하는 비용이 커지고 있는가?

자동화는 문제가 실제로 생겼을 때 도입하는 편이 낫다. 자동화 자체도 관리 비용이 있기 때문이다.

자동화를 고려할 만한 시점은 다음과 같다.

- 토큰 업데이트가 매주 이상 발생한다.
- 수동 복사 때문에 실수나 지연이 반복된다.
- 여러 플랫폼이 가깝게 동기화되어야 한다.
- 토큰 업데이트를 둘러싼 팀 간 조율이 부담스럽다.

---

## 가벼운 자동화부터 시작하기

자동화는 처음부터 완전한 시스템으로 만들 필요가 없다. 먼저 반복 작업과 오류 가능성이 큰 부분부터 줄이면 된다.

초기 자동화 대상은 다음이 적절하다.

- 토큰 파일 복사
- JSON을 플랫폼 코드로 변환
- 잘못된 토큰 값 검증
- 네이밍 규칙 검증

이 단계에서는 가벼운 스크립트와 CI 검증만으로도 효과를 볼 수 있다.

CI(Continuous Integration, 지속적 통합)는 코드 변경이 저장소에 올라올 때 자동으로 빌드, 테스트, 검증을 수행하는 개발 방식이다.

---

## 토큰 검증 추가하기

검증은 토큰 데이터가 정해진 규칙을 만족하는지 확인하는 과정이다.

토큰 검증은 다음 문제를 확인할 수 있다.

- 중복된 토큰 이름
- 잘못된 색상 형식
- 필수 필드 누락
- 일관성 없는 네이밍 규칙
- 잘못된 spacing 단위
- 잘못된 typography 값

검증은 로컬 스크립트로 실행할 수도 있고, pull request가 열릴 때 CI에서 자동으로 실행할 수도 있다.

검증을 먼저 도입하면 잘못된 토큰이 플랫폼 코드로 퍼지기 전에 막을 수 있다.

---

## 플랫폼별 파일 생성하기

검증 다음 단계는 변환이다.

변환은 플랫폼 독립적인 JSON 토큰을 각 플랫폼에서 바로 사용할 수 있는 파일로 바꾸는 작업이다.

예를 들어 다음처럼 변환할 수 있다.

| 플랫폼 | 변환 결과 |
| --- | --- |
| iOS | Swift 코드 |
| Android | XML 또는 Kotlin |
| Web | CSS variables |

---

## 가벼운 자동화 워크플로우

가벼운 자동화 단계에서는 검증과 파일 생성은 자동화하지만, 생성된 파일을 각 플랫폼 저장소에 반영하는 일은 개발자가 직접 할 수 있다.

워크플로우는 다음과 같다.

1. 디자이너가 토큰을 업데이트하고 pull request를 연다.
2. CI가 토큰을 검증한다.
3. 머지된 뒤 개발자가 생성 스크립트를 실행한다.
4. 생성된 파일을 플랫폼 코드베이스에 복사한다.
5. 플랫폼 팀이 변경 사항을 리뷰하고 테스트한다.

이 방식은 자동화와 통제 사이의 균형을 잡는다. 팀은 일관된 생성과 검증의 이점을 얻고, 플랫폼 팀은 변경을 언제 적용할지 직접 결정할 수 있다.

---

## 완전 자동화

완전 자동화에서는 토큰 파일 생성에서 끝나지 않는다. 생성된 결과물이 각 플랫폼 코드베이스로 자동 전달된다.

대표적인 방식은 두 가지다.

1. CI가 각 플랫폼 저장소에 pull request를 연다.
2. CI가 토큰 패키지를 배포하고 앱 팀이 dependency로 사용한다.

이 단계에서는 개발자가 파일을 직접 복사하지 않는다. 대신 자동 생성된 pull request를 리뷰하거나, 토큰 패키지 버전을 업데이트한다.

---

## 방식 1. 플랫폼 저장소에 PR 자동 생성

중앙 토큰 저장소에서 토큰이 변경되면 CI가 다음 작업을 수행한다.

1. 플랫폼별 파일을 생성한다.
2. Android, iOS, Web 저장소를 clone한다.
3. 새 branch를 만든다.
4. 변경된 토큰 파일을 반영한다.
5. 제목과 changelog가 포함된 pull request를 연다.
6. 관련 팀에 리뷰를 요청한다.

이 방식은 코드 생성과 복사를 자동화하면서도, 플랫폼 팀이 변경 적용 시점을 통제할 수 있게 한다.

예를 들어 Android 저장소에는 다음과 같은 PR이 자동으로 열릴 수 있다.

```markdown
# Token Update: v2.4.0

## 변경 사항
- color.background.secondary: #F2F2F7 → #F5F5F5
- spacing.xs 추가
- color.legacy.blue 제거

## 생성된 파일
- res/values/colors.xml
- res/values/dimens.xml
- GeneratedColors.kt
```

플랫폼 팀은 이 PR을 리뷰하고, 필요한 화면을 확인한 뒤 머지하면 된다.

---

## 방식 2. 패키지로 배포

다른 방식은 토큰을 패키지로 배포하는 것이다.

CI가 토큰 변경을 감지하면 새 버전의 토큰 패키지를 릴리즈한다. 앱 팀은 준비되었을 때 dependency 버전을 올린다.

Android에서는 다음처럼 사용할 수 있다.

```kotlin
dependencies {
    implementation("com.example.designsystem:tokens:2.4.0")
}
```

이 방식은 하나의 플랫폼 안에서 여러 앱이 같은 디자인 시스템을 공유할 때 유용하다.

예를 들어 같은 회사에 여러 Android 앱이 있고, 모두 같은 토큰을 사용해야 한다면 토큰을 라이브러리로 배포하는 방식이 안정적일 수 있다.

---

## 문서도 파이프라인에 포함할 수 있다

완전 자동화 단계에서는 문서도 토큰과 함께 업데이트할 수 있다.

예를 들어 CI가 다음 내용을 자동 생성할 수 있다.

- 변경 사항 changelog
- deprecated 토큰 목록
- 이름이 변경된 토큰 목록
- 제거된 토큰 목록
- 토큰 사용 예시

문서가 토큰 JSON에서 생성되면 실제 토큰 값과 문서가 어긋나는 문제를 줄일 수 있다.

---

## 웹과 모바일은 같은 토큰을 공유해야 할까?

모바일 플랫폼끼리는 공통점이 많다. Android와 iOS는 화면 크기, 터치 인터랙션, 앱 내비게이션 같은 면에서 비슷한 문제를 해결한다.

하지만 Web은 다르다.

Web은 모바일 앱과 브랜드 색상을 공유할 수는 있지만, 그 외에는 요구사항이 많이 달라진다.

예를 들어 Web은 다음을 고려해야 한다.

- 넓은 데스크톱 화면
- 반응형 레이아웃
- 마우스 인터랙션
- 사이드바와 내비게이션 메뉴
- 긴 콘텐츠를 위한 타이포그래피 구조

반면 모바일 앱은 터치 중심 인터랙션, 제한된 화면 크기, 세로 스크롤 중심 레이아웃이 많다.

따라서 Web과 Mobile을 억지로 같은 토큰 세트에 넣으면 오히려 복잡성이 커질 수 있다.

공유할 수 있는 것은 공유하되, 다르게 가져가야 하는 것은 분리하는 편이 낫다.

예를 들어 저장소는 공유하되 토큰 파일은 나눌 수 있다.

```
tokens/
  mobile-base.json
  android.json
  ios.json
  web-base.json
```

이렇게 하면 검증 스크립트나 배포 파이프라인은 공유하면서, Web과 Mobile의 토큰 체계는 각자 유지할 수 있다.

---

## 거버넌스도 필요하다

토큰 시스템이 커지면 기술 문제만 남는 것이 아니다. 누가 결정하고, 누가 승인하고, 충돌을 어떻게 해결할지도 정해야 한다.

이를 **거버넌스(governance)**라고 한다. 거버넌스는 시스템을 운영하기 위한 의사결정과 승인 체계를 말한다.

토큰 시스템에서는 다음 질문이 중요해진다.

- 토큰은 누가 변경할 수 있는가?
- 새 토큰은 누가 승인하는가?
- 토큰 값에 대한 의견 충돌은 누가 조율하는가?
- deprecated 토큰은 언제 제거하는가?
- 플랫폼별 override는 누가 관리하는가?
- 시스템이 커질수록 어떤 승인 절차가 필요한가?

초기에는 가볍게 시작해도 된다. 하지만 팀과 플랫폼이 늘어나면 명확한 운영 규칙이 필요하다.

---

## 정리

여러 플랫폼에서 토큰을 운영하려면 수동 핸드오프만으로는 한계가 있다. 플랫폼이 늘어나면 토큰 이름과 값이 어긋나기 쉽고, 변경 사항을 추적하기도 어려워진다.

이를 해결하려면 중앙 토큰 저장소가 필요하다. 토큰은 JSON 같은 플랫폼 독립적인 형식으로 관리하고, Android, iOS, Web은 이를 각자 필요한 코드로 변환해서 사용한다.

토큰을 확장하는 흐름은 다음과 같다.

```
수동 핸드오프
→ 중앙 토큰 저장소
→ JSON 기반 토큰 관리
→ 문서화
→ 토큰 검증
→ 플랫폼별 파일 생성
→ CI 자동화
→ PR 자동 생성 또는 패키지 배포
```

처음부터 완전 자동화를 목표로 할 필요는 없다. 먼저 토큰 저장소와 문서화, 검증부터 시작해도 충분하다.

변경이 많아지고 플랫폼이 늘어나면 자동 생성과 CI를 추가하면 된다. 중요한 것은 모든 팀이 같은 기준으로 토큰을 이해하고, 변경 사항을 추적할 수 있는 구조를 만드는 것이다.

---

## 후기

본문에서는 수동 핸드오프 방식의 한계로, 디자이너가 플랫폼 간 동기화를 유지하기 위한 커뮤니케이션을 떠안게 될 수 있다고 설명한다. 예시로는 다음과 같은 질문이 나온다.

> “토큰 업데이트했나요?”
>
>
> “변경 사항 확인할 수 있게 빌드 받을 수 있을까요?”
>
> “새 토큰은 어느 릴리즈에 포함되나요?”
>

다만 나는 이 질문들이 디자이너의 책임으로 이어지는 것은 적절하지 않다고 생각한다.

디자이너가 확인해야 할 것은 토큰이 코드에 업데이트되었는지 자체가 아니라, 최종 제품에 디자인 의도가 제대로 반영되었는지다. 토큰이 추가되었는지, 제거되었는지, 어떤 코드 파일에 반영되었는지는 디자이너가 코드를 직접 보지 않는 이상 확인하기 어렵다. 따라서 디자이너가 토큰 적용 여부를 추적하기보다는, 완성된 화면을 기준으로 디자인 검수를 하는 것이 더 자연스럽다.

물론 디자이너가 토큰 구조를 이해하고 변경 흐름에 관심을 가지는 것은 도움이 된다. 하지만 토큰 업데이트 여부를 계속 확인하고 추적하는 일이 디자이너의 책임이 되어서는 안 된다. 개발팀은 토큰 변경이 어떤 릴리즈에 포함되는지, 어떤 플랫폼에 반영되었는지 추적할 수 있는 프로세스를 별도로 갖추는 편이 적절하다.

또 하나 의문이 들었던 부분은 “디자이너는 Figma도 토큰 저장소와 동기화해야 한다. 그렇지 않으면 개발자나 토큰 관리자에 의해 변경된 내용이 디자인 파일에는 나타나지 않는다”는 설명이다.

디자인 파일을 최신 토큰과 동기화해야 한다는 점은 이해할 수 있다. 여러 디자이너가 함께 작업한다면 다른 디자이너가 업데이트한 토큰을 import해야 할 수도 있다. 하지만 개발자가 토큰 값을 직접 변경할 수 있어야 하는지는 별개의 문제다.

개발자가 토큰에 대해 의견을 제시하는 것은 필요하다. 예를 들어 네이밍이 코드 구조와 맞지 않거나, 플랫폼 구현상 문제가 있거나, 기존 토큰과 충돌하는 경우에는 개발자의 피드백이 중요하다. 하지만 토큰의 의미와 값에 대한 최종 결정은 디자인 의도를 가장 잘 이해하는 디자이너가 내리는 편이 적절하다고 생각한다.

특히 개발자가 디자이너의 의도를 충분히 확인하지 않은 상태에서 토큰 이름이나 값을 임의로 수정하는 것은 위험할 수 있다. 토큰은 단순한 코드 상수가 아니라 디자인 의도를 담은 결정이기 때문이다. 따라서 개발자는 토큰을 직접 수정하기보다, 변경 제안이나 리뷰 의견을 남기고 디자이너가 최종 판단하는 구조가 더 안전하다.

정리하면, 토큰은 디자이너와 개발자가 함께 관리해야 하지만 역할은 구분되어야 한다. 디자이너는 토큰의 의도, 값, 사용 맥락을 책임지고, 개발자는 구현 가능성, 코드 구조, 플랫폼별 제약을 검토하는 역할을 맡는 것이 적절하다. 공동 소유권은 모두가 같은 일을 한다는 뜻이 아니라, 각자의 전문성을 바탕으로 같은 시스템을 함께 유지한다는 뜻에 가깝다.
