## 도입 — 홀리스틱 마인드셋(Holistic Mindset)

에피그래프: *"얼굴을 스케치하기 전에 속눈썹을 완벽하게 만들지 마라. 먼저 기본 모양을 그리고, 그 다음 디테일을 추가하라."*

지난 챕터(§2)에서 랜드스케이프 그래프를 그려 필요한 도메인을 확립했다. 이 챕터의 목표는 그 그래프를 실제 코드로 바꾸는 것이다. 이때 쓰는 태도가 **홀리스틱 마인드셋** — 뷰·네트워킹·영속성처럼 한 도메인을 처음부터 끝까지 완성한 뒤 다음으로 넘어가는 방식과 정반대로, **작동하는 프로토타입에 도달할 때까지 여러 도메인을 동시에, 얕게 정의하고 연결**하는 접근이다. "코드로 스케치한다"고 생각하면 된다. 이 방식은 디테일을 알아내는 동안에도 속도(모멘텀)를 유지하게 해준다.

## § 3.1 그래프 노드와 코드의 관계

**각 노드는 자체 도메인이며, 그 안에 클래스·데이터 모델·메서드가 어떤 조합으로든 들어갈 수 있다.** 전체 랜드스케이프를 한 번에 늘어놓으면 압도적이므로, 불필요한 부분은 회색 처리하고 지금 끝내야 할 Course 기능에 집중한다. Course의 직접 의존성(Tutor·Calendar·TodoList·Store)에 네트워킹 의존성 하나를 더해, 이 챕터에서 코드로 정의할 도메인 범위를 확정한다.

노드가 "클래스인지 모듈인지"는 아직 결정하지 않아도 된다. Networking은 재시도·백그라운드 동기화까지 갖춘 거대한 라이브러리로 자랄 수도, 작은 헬퍼 함수 하나로 끝날 수도 있다 — 무엇이 되든 자체 도메인 안에 있다는 사실만 중요하다. 이런 도메인 중 일부는 책 후반에 가서야 별도 네임스페이스·모듈로 추출된다.

## § 3.2 인터페이스와 API에 대한 짧은 메모 — 용어 정리

이 챕터에서 "API"는 백엔드 API가 아니라 `courseService.loadCourses()`처럼 **타입이 어떻게 사용되는지**를 뜻하고, "인터페이스"는 사용자 인터페이스가 아니라 (Swift `protocol`, Kotlin `interface` 같은) **인터페이스 타입**을 뜻한다.

핵심 주장: **이 챕터에서는 인터페이스 타입을 전혀 쓰지 않고 구체 구현(concrete implementation)만으로 진행한다.** 일부 개발자는 인터페이스로 너무 일찍 시작하는데, 이 챕터는 인터페이스 없이도 얼마나 멀리 갈 수 있는지 보여준다. 인터페이스는 이후 챕터(§4.4.6, DI 파트)에서 깊이 다룬다.

## § 3.3 무엇을 우선시할 것인가 — 위에서부터, UI는 미룬다

**경험칙: 가장 높은(상위) 도메인을 가장 높은 우선순위로 간주한다.** Course 기능을 브리핑 받았다면 Course가 랜드스케이프에서 가장 높은 도메인이므로 거기서 시작한다. 사용자 입장에서는 Course가 내부적으로 Store에 의존하든 Networking을 쓰든 중요하지 않다 — 오히려 잘 작동하는 Calendar 도메인보다 미완성이라도 Course 기능이 있는 편이 더 가치 있다.

**§ 3.3.1 UI는 지연한다.** Course부터 시작한다고 UI 구현부터 시작하는 게 아니다. 이 단계의 랜드스케이프는 **모델 계층 또는 비즈니스 로직 계층**으로 간주된다. 나중에 Course를 사용하는 CourseUI 도메인이 그 위에 생기겠지만(Figure 3.4), CourseUI와 Course는 별개 도메인이면서도 같은 모듈에 살 수 있다. UI를 의도적으로 뒤로 미루는 이유와 장단점은 다음 챕터(§4) 도입부에서 다룬다.

**§ 3.3.2 CourseUI는 Course의 내부를 신경 쓰지 않는다.** CourseUI 관점에서는 Course가 거대한 갓 클래스든 400개의 미니 함수든 상관없다. CourseUI가 Course에게 요구하는 건 "코스를 줘"와 "TodoItem을 체크오프하고 있어" 정도뿐이다 — 이것이 다음 절의 **호출부 설계(call-site design)** 원칙으로 바로 이어진다.

## § 3.4 Course 도메인 구현 — 호출부에서 시작해 플레이스홀더로 컴파일 유지

핵심 원칙: **내부 구현보다 먼저 호출부(call-site)에서 "어떻게 사용하고 싶은가"를 설계한다.** API가 견고하면 호출부를 건드리지 않고도 내부 구현을 마음껏 조정할 수 있고, 명명에도 도움이 되며 구현 디테일에 산만해지는 것도 막아준다. UI를 만들 동료는 Course의 내부 구현이 아니라 "쓰기 쉬운가"만 신경 쓴다는 비유가 이를 잘 보여준다.

**§ 3.4.1 코스 가져오기.** `courseService.loadCourse(id:)`로 `Course`를 가져오는 코드를 먼저 작성한다 — 이 시점엔 아무것도 컴파일되지 않는다. 이것이 의도된 흐름이다: 호출부를 먼저 설계하고, 그다음 컴파일되는 프로그램이 될 때까지 필요한 것들을 채워 넣는다. 이 단계에서 `CourseService`의 내부 동작이나 의존성은 전혀 모른 채로 진행한다 — 네트워킹·테스트·DI를 생각할 필요가 없어진다.

**§ 3.4.2 Course 데이터 모델.** `id`, `tutor`, `schedule([TodoItem])`, `calendarEvent(옵셔널)` 네 속성을 가진 작은 구조체. `schedule`은 완료 표시를 위해 `var`(가변)로 둔다.

**§ 3.4.3 플레이스홀더 사용.** `Course.placeholder` 같은 정적 샘플 값을 도입해 스케치 중 다른 코드를 깨끗하게 유지한다. **플레이스홀더 접근법이 홀리스틱 개발의 핵심** — 전체 구조를 확립하면서 구현 디테일은 지연시킨다.

**§ 3.4.4~3.4.5 Tutor 모델과 invalid state 제거.** Tutor에 `callOut: String?`과 `isCalloutDismissed: Bool`을 따로 두면 "콜아웃이 nil인데 무시 여부는 false"처럼 **말이 안 되는 상태 조합**이 가능해진다. 해결책은 `enum Callout { case message(String); case dismissed }`를 도입해 `callOut: Callout?` 하나로 합치는 것 — 이제 nil이거나, 메시지가 있거나, 무시됐거나 셋 중 하나만 가능해 애초에 불가능한 상태를 만들 수 없다.

**§ 3.4.6~3.4.7 TodoItem — "더 풍부한 데이터"로 미래 유연성 확보.** 직관적으로는 `isChecked: Bool`을 쓰고 싶지만, 대신 `fulfillmentDate: Date?`를 저장하고 `completed`를 **계산 속성**(get/set 둘 다 가진)으로 파생시킨다. 같은 결과(체크 여부)를 얻으면서도 "언제 체크됐는가"라는 더 풍부한 정보를 공짜로 얻어, 나중에 요구사항이 바뀌어도 데이터 모델을 리팩터링할 필요가 없다. 별도로 **"isChecked"라는 이름 자체가 UI 관점의 이름이라 부적절하다**는 지적도 있다 — 데이터 세계에서는 UI 표현(체크마크·토글·라벨)과 무관한 `completed` 같은 이름을 써야, 시각적 구현이 데이터 모델 이름에 새어 들어가지 않는다. 이 둘은 챕터 요약(§3.10)에서 **서로 다른 두 원칙**으로 분리된다 — "미래 유연성을 위한 풍부한 데이터"의 예시가 `fulfillmentDate`이고, "UI 관점이 아닌 데이터 관점 명명"은 별개 항목이다.

**§ 3.4.8 course.schedule.weekly / .daily.** `Array<TodoItem>`을 확장해 계산 속성 `weekly`/`daily`를 제공한다. Swift 한정 문법 설탕이며(다른 언어라면 `Course`에 `weeklySchedule`/`dailySchedule` 속성을 직접 두면 된다), 이 시점엔 실제 날짜 필터링 로직 없이 `self`를 그대로 반환하는 플레이스홀더로 컴파일만 유지한다.

**§ 3.4.9 CalendarEvent.** `date: Date`와 `link: URL?` 두 필드뿐인 작은 구조체. 실제 스케줄러(통화 예약 UI)를 만드는 대신, 이미 잡힌 일정에 대한 읽기 전용 정보만 다룬다 — "Join call" 버튼은 지금은 클로저 하나만 트리거하면 되고, 딥링크 여부는 UI 단계에서 결정한다.

**§ 3.4.10 지연 == 게으름?** 저자는 이 지연을 "게으름"이 아니라 **"주된 목표를 시야에서 잃지 않는 것"**으로 규정한다. 딥링크 엔진이나 오프라인 동기화 같은 디테일에 빠지면 Course 기능 자체가 없는 채로 시간을 다 쓸 위험이 있다. 지금은 CourseService를 마무리해, 미완성이라도 다른 사람이 통합·사용할 수 있게 하는 것이 우선이다.

## § 3.5 CourseService 정의 — 인공 지연으로 비동기 이슈 조기 발견

`loadCourse(id:)`를 가진 `CourseService` 클래스를 정의하되, **네트워크·DI·테스트는 전혀 고려하지 않고 컴파일만 되게 만드는 것**이 목표다. 실제 네트워크 호출 대신 `Task.sleep(for: .seconds(2))`로 인공 지연을 넣고 `Course.placeholder`를 반환한다.

이 인공 지연이 단순한 눈속임이 아닌 이유 — 동료가 이미 이 서비스를 통합해 "호출 후 로더를 숨긴다" 같은 UI를 짜고 있다면, 호출이 즉시(동기처럼) 끝나버리면 로딩 상태를 검증할 기회조차 없어진다. **비동기 호출을 실제처럼 흉내 내야 비동기 관련 버그를 이 이른 단계에서 이미 잡아낼 수 있다.**

## § 3.6 Store 컴포넌트 설계 — 순진하지만 진짜인 API

Course는 오프라인 지원을 위해 Store에 의존한다(Figure 3.9, 점선 — 아직 세부는 모름). 오프라인 지원은 동기화 알고리즘·델타 해결·버전 충돌까지 함의하는 위협적인 주제지만, 홀리스틱 접근으로 **메모리 저장소**로 시작해 API 설계에만 집중한다.

**§ 3.6.1 다시 호출부에서 시작.** 원래 `loadCourse(id:)`였던 것을 "최신 데이터를 강제로 가져온다"는 의미의 `loadFresh(id:)`로 이름을 바꾸고, 새 `loadCourse(id:)`는 "로컬 저장소 우선 조회 → 없으면 loadFresh 폴백" 역할을 맡는다. `loadFresh`는 `CourseService`만 호출하면 되므로 `private`으로 — **타입의 API 표면을 작게 유지하는 것**이 좋은 설계 습관이라는 원칙이 여기서 나온다.

**§ 3.6.2~3.6.3 Store\<T\> 구현과 get() 시그니처 6단계.** `Store<T>`는 내부에 `[UUID: T]` 딕셔너리 하나만 갖는다. `get(identifier:load:)`의 시그니처는 호출부 요구를 반영해 단계적으로 완성된다:

| 단계 | 시그니처 변화 | 이유 |
|---|---|---|
| 1 | `load: () -> T` | 기본형 |
| 2 | `load: () throws -> T` | 클로저가 오류를 던질 수 있어야 함 |
| 3 | `load: () async throws -> T` | 클로저가 비동기이기 때문 |
| 4 | `get(identifier:load:) -> T` | 매개변수 확정 |
| 5 | `get(...) async -> T` | load가 비동기이므로 get도 비동기 |
| 6 | `get(...) async rethrows -> T` | **rethrows** — Store는 load 내부에서 무슨 일이 일어나는지 모르므로, 자신이 직접 오류를 던지는 `throws`가 아니라 클로저의 오류를 그대로 재전파하는 `rethrows`를 쓴다 |

최종 구현은 딕셔너리에 값이 있으면 반환, 없으면 `load()`를 호출해 저장 후 반환하는 단순한 캐시다.

**§ 3.6.5~3.6.6 의도적으로 순진한 구현, 그러나 낮은 우선순위.** 데이터가 오래되면? 부분 업데이트는? 경쟁 조건은? 스레드 안전성은? — 문제는 뻔히 보이지만 **지금 모든 걸 완벽히 구현하는 것보다 전체를 연결하는 게 더 중요**하다. 중요한 건 `CourseService`가 Store를 올바른 API로 사용하고 있다는 사실이며, 내부 구현은 API 경계 뒤에 숨겨져 있어 CourseService 입장에서는 "완전해 보인다."

> **Store는 플레이스홀더가 아니라 진짜 컴포넌트다.** `Course.placeholder`(정적 샘플 *값*의 지연)와 `Store<T>`(실제 동작하는 클래스이지만 내부 저장 *메커니즘*이 메모리 딕셔너리로 단순화된 것)는 "디테일을 지연한다"는 철학은 같아도 지연 대상이 다르다 — 하나는 값, 하나는 구현.
>
> **Store 업그레이드와 새 기능은 독립적인 타임라인이다.** Store는 그래프상 Course보다 아래에 있어 Course를 전혀 모른다. 따라서 "Course가 끝나야 Store가 완성된다"는 인과관계는 없다. 실제 오프라인 요구사항이 생겼을 때 누군가 Store 내부(예: SQLite로 교체)를 업그레이드하면, `Store<Course>`든 나중에 생길 `Store<B>`든 호출 코드 한 줄 바꾸지 않고 자동으로 혜택을 받는다.

## § 3.7 컴포넌트 재사용 시의 트레이드오프 — Store를 제네릭으로 만든 이유

"3번 쓸 때까지 재사용 가능하게 만들지 마라"는 격언에 대한 **반례**. `CourseStore`처럼 특정 타입 전용으로 만들 수도 있었지만, 랜드스케이프에서 Store는 Tutor·Course 도메인 **바깥**에, 그보다 **아래** 위치한다(Figure 3.10) — Store는 위쪽 도메인을 알 수 없는 방향성을 갖는다.

핵심 논지: **"나중에 유용할 것 같아서" 제네릭으로 만드는 게 아니다** — 그건 오버엔지니어링 함정이다. Store를 제네릭으로 만드는 이유는 오직 하나, **Store가 자체 도메인에 있고 기초(foundational) 컴포넌트로 봉사하기 때문**이다. Store는 비트와 바이트만 다룰 뿐 Course에 대해 알 이유가 없다 — 도메인 위치가 제네릭화를 정당화하는 유일한 근거다.

## § 3.8 최종 결과 — 컴파일되고 실행되는 프로그램

지금까지 만든 타입들을 타입 수준 다이어그램(Figure 3.11)으로 정리하면, UI가 구현할 준비가 된 Course 도메인이 완성돼 있다. 실제로 `courseService.loadCourse(id:)`를 호출해 `Tutor`·`schedule`·`calendarEvent`가 출력되는 것을 확인한다 — **"컴파일되고 실행된다"는 것 자체가 지금까지의 진행을 검증하는 체크포인트**다.

## § 3.9~3.10 결론 및 챕터 요약

한 타입을 깊이 파는 대신 (불)완전한 구현으로 필요한 모든 타입을 함께 붙였다. **연필과 종이로 얼굴을 스케치하는 것과 같다** — 콧구멍부터 완벽하게 그리면 나중에 머리 전체 비율이 어긋나 다시 그려야 할 위험이 있다. 시스템 디자인 인터뷰에서도 한 컴포넌트에 과몰입하는 것보다 전체 상호작용과 데이터 흐름을 보여주는 게 중요하다는 비유로 이어진다. GraphQL처럼 낯선 기술이 필요할 때도, API부터 설계하고 디테일은 나중에 구현할 수 있다는 자신감이 있으면 덜 위협적이다.

**챕터 요약 4대 원칙**

| 원칙 | 핵심 |
|---|---|
| **API 우선 컴포넌트 설계** | 인체공학에 집중하려면 내부 구현 전에 호출부부터 설계. 좋은 API에 인터페이스(프로토콜)는 필요 없다 — 구체 구현으로 충분 |
| **견고한 데이터 모델링** | 불가능한 상태를 방지하는 강한 타입 (`Callout` enum). 같은 결과를 내는 더 풍부한 데이터 선택 (`fulfillmentDate` vs `isChecked`). UI가 아닌 데이터 관점으로 명명해 시각적 구현이 이름에 새어들지 않게 함 |
| **플레이스홀더 주도 개발** | 컴파일·실행될 때까지 종교적으로 플레이스홀더 사용. 비동기 플레이스홀더엔 인공 지연을 넣어 비동기 이슈를 조기 발견. 매 반복마다 플레이스홀더를 한 단계씩만 더 깊이. 플레이스홀더는 해당 컴포넌트의 우선순위 자체를 낮춘다 |
| **의존성 계층을 통한 홀리스틱 작업 + 기초 컴포넌트 제네릭화** | 가장 높은 우선순위 도메인부터 톱다운으로. 저수준 구현의 완벽함보다 고수준 도메인의 완성을 우선. 인프라 컴포넌트가 기초 도메인일 땐 제네릭으로 (`Store<T>` vs `CourseStore`) — 기초 컴포넌트는 도메인 중립적이어야 함 |

## 나누고 싶은 이야기

### 1. Compose/Kotlin으로 옮기면 — Store\<T\>는 제네릭 캐시, Callout은 sealed interface

이 챕터의 Swift 코드는 Compose 생태계로 거의 1:1 대응된다.

- **`Store<T>`** → Kotlin 제네릭 클래스로 그대로 옮길 수 있다. `get(identifier:load:)`의 6단계 시그니처 진화(throws → async → rethrows)는 Kotlin에서는 `suspend fun get(identifier: UUID, load: suspend () -> T): T`로 훨씬 단순해진다 — Kotlin의 `suspend` 함수는 기본적으로 예외를 그대로 전파하므로 `rethrows` 같은 별도 키워드가 필요 없다. 다만 §3.6.3에서 저자가 강조한 "메서드 시그니처는 유지보수자만 신경 쓰면 되고, 호출부가 쉬운 게 더 중요하다"는 원칙 자체는 동일하게 적용된다.
- **`Callout` enum (invalid state 제거)** → Kotlin `sealed interface Callout { data class Message(val text: String): Callout; data object Dismissed: Callout }`. `callOut: Callout?` 하나로 "없음/보임/무시됨" 세 상태를 배타적으로 표현하는 패턴은 Compose UI 상태 모델링(`UiState` sealed class)에서 반복적으로 쓰는 기법과 정확히 같다.
- **`completed` 계산 속성 (get/set)** → Kotlin의 custom property getter/setter로 그대로 대응: `var completed: Boolean get() = fulfillmentDate != null; set(value) { fulfillmentDate = if (value) Instant.now() else null }`.
- **`Array<TodoItem>.weekly/.daily` 확장** → Kotlin extension property `val List<TodoItem>.weekly: List<TodoItem> get() = ...` 로 동일하게 가능. Swift만의 문법 설탕이 아니라 Kotlin도 지원하는 패턴이라, 저자가 "다른 언어에서는 Course에 직접 속성을 두라"고 한 대안보다 오히려 원래 방식 그대로 가져갈 수 있다.
- **`Task.sleep`로 인공 지연** → `delay(2000)` (kotlinx.coroutines)로 동일한 역할.
- **호출부 설계 원칙** → Compose에서는 이게 "ViewModel/UseCase의 public 함수 시그니처를 Composable이 어떻게 부를지 먼저 스케치한다"는 뜻이 된다. CourseUI가 Course의 내부(§3.3.2)를 몰라도 되는 것처럼, `@Composable`이 `ViewModel`의 내부 구현(어떤 `Repository`를 쓰는지)을 몰라도 되게 설계하는 것과 같은 원칙이다.

### 2. Pillo 앱에 대입하면 — Store\<T\>가 DrugBank·FDA·Pillo 통합의 캐시 계층이 될 수 있다

Pillo에서 DrugBank·FDA·Pillo 자체 API 세 소스를 다루는 상황에 이 챕터의 Store 패턴이 그대로 적용된다. 각 소스별로 `DrugBankCache`, `FdaCache`처럼 따로 만들고 싶은 유혹이 있겠지만, §3.7의 논리를 따르면 캐시 자체는 "약물 도메인을 몰라도 되는" 기초 컴포넌트이므로 `Store<DrugInfo>`, `Store<InteractionResult>`처럼 **하나의 제네릭 캐시 클래스를 도메인별로 인스턴스화**하는 편이 낫다. 세 API의 결과가 최종적으로 같은 모양(`DrugInfo` 같은 통합 모델)으로 정규화된다면, 캐시 계층은 그 정규화된 타입 하나만 알면 되고 어느 소스에서 왔는지는 몰라도 된다.

또한 §3.4.3~3.4.10의 "플레이스홀더 주도 개발"은 지금 Pillo 개발에 바로 쓸 수 있는 실무 전략이다 — 세 외부 API 연동이 아직 안정화되지 않았더라도, `DrugInfo.placeholder`로 UI와 도메인 로직을 먼저 컴파일·실행 가능한 상태로 만들고, 각 API 연동은 이미 확정된 `DrugRepository` API 뒤에서 나중에 하나씩 교체해 넣을 수 있다. Store 부분의 "구현 업그레이드와 새 기능은 독립적 타임라인"이라는 supplement 원리도 마찬가지 — DrugBank 연동이 실제 네트워크 호출로 바뀌는 시점과 Pillo의 다른 기능(예: 복약 알림) 개발 완료 시점은 서로 인과관계가 없다.

### 3. 오늘의 멘탈 모델 업데이트 — "호출부 우선"은 전 챕터를 관통하는 하나의 규율

이 챕터에서 반복적으로 등장하는 패턴은 사실 하나다: **매번 상위 계층이 무엇을 원하는지부터 정하고, 그 요구를 그대로 하위 계층의 시그니처로 번역한다.** `loadCourse` → `Course` 모델 → `CourseService` → `Store` → `get()` 시그니처까지, 각 단계는 이전 단계의 호출부가 요구하는 형태를 그대로 받아 적은 것이지 하위 계층에서 "이렇게 설계하면 재사용성이 좋겠다"는 식으로 위로 올라온 게 아니다. §3.6.3의 get() 시그니처가 `throws`에서 `async throws`를 거쳐 `rethrows`로 완성되는 과정도, 매번 "CourseService가 이걸 어떻게 부를 것인가"를 먼저 묻고 답한 결과다.

퀴즈에서 실제로 헷갈렸던 지점들(`Store<T>`와 `Store<Course>`가 별개 클래스라는 오해, get()의 마지막 키워드를 `throws`로 착각한 것)도 결국 이 원칙을 놓쳤을 때 생기는 오류다 — Store는 항상 "T"라는 하나의 미지수로 존재하고, CourseService라는 특정 호출부가 그 자리에 `Course`를 채워 넣을 뿐이라는 점, 그리고 매 키워드 추가가 항상 "호출부가 이미 하고 있는 일"을 뒤늦게 시그니처에 반영하는 것뿐이라는 점을 기억하면 이 챕터의 세부 단계들이 암기가 아니라 하나의 논리로 재구성된다.
