### § 3.1 Spacing — 가장 간과되는 기초

저자는 흥미로운 관찰로 시작한다. 모바일 개발자는 아이콘과 그림자는 잘 챙기지만 *간격 시스템* 은 흔히 간과한다. 디자인을 받으면 픽셀을 측정해 임의의 값을 그대로 박는다. `VStack(spacing: 12).padding(20)` 같은 식.

이게 *매직 넘버* 다. 한 화면에선 멀쩡해도, 시간이 지나며 누군가는 12, 누군가는 14, 누군가는 15 를 쓴다. 그러다 디자이너가 *"전반의 간격을 좀 더 넉넉하게 하고 싶다"* 라고 하면 — 모든 12, 14, 15 의 인스턴스를 수동으로 찾아 바꿔야 한다. 지루하고 오류가 잦다.

#### 1층 — Spacing 프리미티브 (고정 스케일)

해법은 일단 고정 스케일을 잡는 것.

```
4, 8, 16, 24, 32, 40, 48, 64, 80
```

예측 가능한 스케일로 선택을 제한해 *임의의 결정을 제거하고 일관성을 강제* 한다. 공통 그리드와 정렬되는 4·8 기반. 숫자가 커질수록 정밀 제어 필요가 줄어드는 점도 의도된 것이다 (큰 간격은 비례·균형 유지가 더 중요).

```swift
struct Space {
    static let s4: CGFloat = 4
    static let s8: CGFloat = 8
    // ... s80 까지
}
```

> **Note** — Swift 가 숫자로 시작하는 프로퍼티명을 허용하지 않아 `s` prefix. 80 에서 멈추는 이유는, 그 이상의 *큰 간격은 구조적* 으로(중앙 정렬, 컨테이너 래핑) 다뤄지기 때문.

기존 코드의 12 는 스케일에 없다. 이건 *12 가 시스템에 어울리지 않는다는 신호*. `Space.s8` 또는 `Space.s16` 중 가까운 값으로 보낸다. 확신이 없으면 디자이너와 상의.

#### 2층 — Semantic Spacing

프리미티브를 직접 쓰는 건 컬러·폰트와 같은 문제를 갖는다. `Space.s16` 은 그저 값일 뿐, *왜* 이 간격을 쓰는지 말해주지 않는다.

```swift
struct Spacing {
    static let xxs = Space.s4
    static let xs  = Space.s8
    static let sm  = Space.s16
    static let md  = Space.s24
    static let lg  = Space.s32
    // ...
}

VStack(spacing: Spacing.md) { ... }.padding(Spacing.lg)
```

이득은 두 가지. 의미가 붙는다(*"이건 lg 패딩이구나"*). 변경이 중앙화된다 — `Space.s24` 를 `Space.s22` 로 옮기려면 모든 콜 사이트가 아니라 *`Spacing.md` 한 곳* 만 고치면 끝.

#### 3층 — Dynamic Spacing

정적 시멘틱 값의 한계는, 폰과 태블릿이 같은 간격을 받는다는 점. 두 버튼 사이가 폰에선 *타이트*(`Spacing.sm`), 태블릿에선 *느슨*(`Spacing.md`)일 필요가 있을 수 있다.

뷰마다 size class 분기를 적으면 반복 지옥. 그래서 라이브러리 내부로 끌어들인다.

```swift
struct Spacing {
    let value: CGFloat
    let offset: CGFloat

    func dynamic(for sizeClass: UserInterfaceSizeClass?) -> CGFloat {
        sizeClass == .regular ? value + offset : value
    }

    static let md = Spacing(value: Space.s24, offset: Space.s8)
    // ...
}

VStack(spacing: Spacing.md.dynamic(for: sizeClass)) { ... }
.padding(Spacing.lg.value)
```

동적·정적을 같이 노출. `margin(for:)`, `padding(for:)` 같은 헬퍼로 더 확장 가능. 핵심은 *뷰가 브레이크포인트를 결정하는 부담을 라이브러리가 흡수* 한다는 것.

### § 3.2 Icons — 시멘틱과 프리미티브의 *공존* 허용

아이콘은 컬러와 다르다. 이게 Ch.3 의 핵심 학습이다.

#### 시멘틱 아이콘

```swift
struct Icons {
    static let close   = UIImage(named: "cross")
    static let dismiss = UIImage(named: "cross")
    static let remove  = UIImage(named: "cross")
}
```

세 시멘틱 아이콘이 같은 `cross` 자산을 쓴다. 이득은 컬러와 동일. *의도* 가 코드에 박힌다(`Icons.dismiss` 한 줄로 "토스트 닫기" 임을 안다). 디자이너가 `remove` 만 *trashcan* 으로 바꾸겠다고 하면 — `cross` 의 다른 사용처에 영향 없이 매핑만 갈아끼우면 끝.

> **Note** — *프리미티브 아이콘 이름이 시멘틱 사용을 누설하지 않도록* 주의. 자산이 `"cross"` 라면 OK. `"dismiss"` 라면 같은 아이콘을 `remove` 로 쓸 때 혼란이 생긴다.

#### 컬러와 다른 점

컬러는 *프리미티브 직접 사용을 최소화* 하라고 했다. 아이콘은 *둘 다 노출* 해도 된다. 왜?

> *"컬러는 전체 팔레트와 모든 시멘틱 컬러를 미리 정의 가능하지만, 아이콘은 모든 시멘틱 사례를 예측하기 어렵다."*

이 한 문장이 핵심. 일회성 아이콘, 마케팅 배너, 실험적 사용처가 끊임없이 들어온다. *모든 사례를 미리 시멘틱화하려는 시도가 더 큰 비용을 만든다*. 그래서 아이콘 시스템은 시멘틱 + 프리미티브 *공존* 을 허용한다.

```swift
struct Icons {
    static let close          = .cross
    static let dismiss        = .cross
    static let remove         = .cross
    static let navigationBack = .arrowBack

    // 프리미티브도 같이 노출
    static let cross     = UIImage(named: "cross")
    static let arrowBack = UIImage(named: "arrow_back")
}
```

*시멘틱은 일관성을 강제하고, 프리미티브는 일회성·변하는 요구의 안전망* 이다.

### § 3.3 Shadows — 시멘틱만으로 충분

그림자는 패턴이 또 다르다.

```swift
struct ShadowStyle {
    static let small  = ShadowStyle(radius: 4,  x: 0, y: 2, opacity: 0.1)
    static let medium = ShadowStyle(radius: 8,  x: 0, y: 4, opacity: 0.2)
    static let large  = ShadowStyle(radius: 16, x: 0, y: 8, opacity: 0.3)

    let radius: CGFloat
    let x: CGFloat
    let y: CGFloat
    let opacity: Double
}
```

저자는 *세 강도(small/medium/large)* 면 대부분의 기능을 지원한다고 주장한다. 그리고 *프리미티브를 따로 노출할 필요가 없다* 고 결정한다. 이유는 단순. *개발자가 그림자의 raw 값을 직접 입력할 일이 드물기 때문*.

대신 escape hatch 는 남긴다. 부족하면 `ShadowStyle` 을 커스텀 값으로 초기화할 수 있다. 단, 네이티브 `shadow()` 가 아니라 우리 `applyShadow()` 를 통하도록 — 다크 모드 처리 같은 부가 로직을 받기 위해.

#### 다크 모드 — *완전히 숨김*

그림자가 다크 UI 에선 묘사가 큰 의미가 없다는 관찰이 흥미롭다. 미묘한 *글로우* 로 바꿀 수도, *완전히 생략* 할 수도 있다. 저자는 후자를 권한다.

```swift
struct ShadowModifier: ViewModifier {
    let style: ShadowStyle
    @Environment(\.colorScheme) private var colorScheme

    func body(content: Content) -> some View {
        if colorScheme == .light {
            content.shadow(/* ... */)
        } else {
            content // 다크 모드에선 그림자 자체를 렌더링하지 않음
        }
    }
}
```

*적응형 컬러 그림자보다 아예 렌더링하지 않는 게 더 깔끔* 하다는 판단.

#### 린팅으로 오용 방지

`applyShadow()` 가 있어도 네이티브 `shadow()` 를 쓰는 개발자가 나온다. 빌드 중 경고를 띄우는 린팅을 같이 깐다. 이게 Ch.4 의 큰 주제로 이어진다.

### § 3.4 결론 — 두 추상화로 사고하기 + 분리 휴리스틱

폰트·컬러·간격·아이콘·그림자는 빙산의 일각. 책이 짚는 *남은 UI 속성들* — 테두리, 모서리 둥글기, 불투명도, 고도, 블러, 그리드, 레이아웃, z-index, 모션. 같은 *프리미티브 vs 시멘틱* 사고를 그대로 적용할 수 있다.

#### 분리 휴리스틱 — 두 질문

Ch.3 까지 다룬 요소들을 정리하면 패턴이 보인다.

| 요소 | 직접 입력? | 사례 예측? | 결과 |
|---|---|---|---|
| Spacing | YES | — | A (둘 다 노출) |
| Icons | — | NO | A (둘 다 노출) |
| Shadows | NO | YES | B (시멘틱만) |
| Colors | NO | YES | B (시멘틱만) |

두 질문이면 끝난다.

```
Q1. 개발자가 raw 값을 직접 입력할 일이 있는가?
    YES → A (둘 다)
    NO  → Q2 로

Q2. 시멘틱 사례를 미리 다 예측 가능한가?
    YES → B (시멘틱만)
    NO  → A (둘 다, 프리미티브는 안전망)
```

새 요소도 같은 룰로 결정. Border width 는 가끔 raw 직접 → A. Corner radius 는 Spacing 과 유사 → A. Elevation level 은 0~5 예측 가능 → B. Z-index 는 직접 쓸 일 없음 → B.

> 가장 흔한 함정은 *"일관성을 위해 항상 시멘틱만"* 또는 *"안전하게 무조건 둘 다"*. 둘 다 잘못이다. Spacing 에 프리미티브 escape 가 없으면 *"이 한 곳만 22pt"* 요청에 우회밖에 답이 없다. Shadows 에 raw 프리미티브를 노출하면 시멘틱 가치가 증발한다. *A 패턴에서도 주력은 시멘틱* 이라는 점이 핵심.

## Scale-04 · Migrating to Semantic UI (실용 챕터 + *심리* 챕터)

> *"코드베이스 마이그레이션은 사람들이 살고 있는 동안 집을 리노베이션하는 것과 같습니다."*

이번 챕터는 두 얼굴을 갖는다. 절반은 기술 — 린터, 스크립트, 어댑터. 나머지 절반은 *심리학*. 이 둘이 분리될 수 없다는 게 책의 주장이다.

### § 4.1~4.3 도입 — 다른 엔지니어의 입장에서 보기

Ch.2~3 에서 깔끔한 시멘틱 UI 를 만들었다. 디자이너는 신이 났다. 자신만만한 발표를 한다.

> *"Semantic UI is the future!"*

박수를 기다린다. 그런데 *침묵*. 한숨 몇 번. 조용한 불평. 무슨 일이?

저자는 잠시 입장을 바꿔보라고 한다. *지금 당신은 기능 엔지니어다*. 어려운 새 기능을 출시하려고 몇 주째 갈아넣고 있다. 마감은 다가오고 긴장은 고조된다. 그런데 갑자기 동료가 새 UI 시스템을 만들었다. 멋져 보인다. 그러나 이제 *모든 걸 마이그레이션* 해야 한다. *지금 필요한 게 그게 아니다*.

더 나쁘게, 곧 출시할 기능 코드는 *완전히 새것* 인데 — 새 UI 시스템이 있다는 이유로 *레거시 UI* 로 라벨된다. 정말 맥 빠진다.

#### 마이그레이션의 실제 모습

> *"인라인 값을 시멘틱 값으로 교체. 그러나 모든 게 한 줄 교체는 아니다."*

```swift
// Before
view.backgroundColor = UIColor.white
label.textColor = UIColor(red: 0.2, ...)

// After
view.backgroundColor = Color.primaryBackground
label.textColor = Color.textPrimary
imageView.image = Icon.warning
```

그림자는 *3 줄(shadowOpacity, shadowOffset, shadowRadius)* 이 `view.applyShadow(Shadow.medium)` *한 줄* 로 줄어든다. 텍스트 스타일은 *5 줄 multi-line* 이 `Text("Account Settings").companyFont(.body)` 한 줄로. 프로젝트에 따라 *수천 줄* 이 이런 식이다.

### § 4.4 팀 준비 — *수도꼭지부터 잠근다*

마이그레이션을 시작하기 전, *새 레거시 UI 가 추가되는 것부터 멈춰야* 한다. 안 그러면 끝없이 따라잡기. 저자가 인용하는 네덜란드 표현이 잘 와닿는다.

> *"수도꼭지가 흐르는 동안 바닥을 닦는 것 (Mopping the floor while the tap is still running)."*

#### § 4.4.1 팀 교육

먼저 팀을 *교육*. 어떤 변화가 오는지, 언제부터 적응이 필요한지 — *지금은 강제가 아니지만 새 코드에선 선호된다* 고 명확히. 풀 마이그레이션을 강요하지 말고 탐색을 초대. *작고 접근하기 쉬운 단계* 부터 — 새 시멘틱 컬러 하나, 새 그림자 한 번.

> **Note** — 스포일러: 나중엔 분명히 레거시 UI 를 *블록* 할 것이다. 하지만 *아무도 옛 시스템을 썼다고 처벌받았다고 느껴서는 안 된다*.

#### § 4.4.2 다가가기 쉽고 도움이 되는 사람 되기

*문서를 작성* — 새 시멘틱 UI 자체에 대한 것뿐 아니라 *마이그레이션 가이드*, *변경 전후 예시* 까지. 프레젠테이션을 하고, Slack 채널을 열고, 다른 사람의 마이그레이션을 도울 *시간 예산을 따로* 잡는다. 사람들이 막히고, 도움이 필요하고, 버그를 발견하고, *누락된 UI 요소* 를 발견할 것이다.

### § 4.5 새 코드가 새 UI 를 쓰도록 보장 — *린터*

PR 마다 잔소리꾼이 되지 말고 *툴링* 을 쓴다.

#### § 4.5.1 린터 메시지의 *문구*

라우 컬러나 raw 간격 같은 *옛 습관이 새 작업에 스며드는 것* 을 막는다. 다만 메시지가 중요하다.

> *"경고 메시지가 단지 잘못을 지적하는 것이 아니라, 도움이 되거나 관련 문서로 링크하도록 만드세요. 문구의 중요성을 과소평가하지 마세요. 이는 메시지가 처벌이 아닌 도움 가이드처럼 느껴지게 합니다."*

> **Note** — *마이그레이션의 심리는 중요하다*. 개발자가 이전에 *"레거시 UI"* 를 쓴 건 *그들의 잘못이 아니다*.

#### § 4.5.2 Soft rollout

처음엔 *경고(error 아님)*. 컴파일·머지 가능. 기대치만 알린다. 팀이 익숙해진 뒤 *CI 에서 에러로 승격*.

#### § 4.5.3 Diff 만 검사

전체 코드베이스에 린트를 돌리면 수백 개의 레거시 위반이 한꺼번에 뜬다. 도구가 *시끄럽고 무의미* 하게 느껴진다. 해법은 *PR 의 diff 에 대해서만* 린트.

#### § 4.5.4 회귀 방지 — 폴더별 룰

soft rollout 으로도 회귀가 일어난다. ChatView 가 시멘틱으로 마이그레이션됐는데, 어느 날 한 dev 가 급해서 *레거시 컬러* 를 다시 박는다. *뷰가 한 번 마이그레이션되면 마이그레이션 상태로 머물러야 한다*.

기능을 `LEGACY_UI` 와 `SEMANTIC_UI` 폴더로 분리한다. LEGACY_UI 의 뷰는 *경고*, SEMANTIC_UI 의 뷰는 *에러*. 완전히 마이그레이션되면 SEMANTIC_UI 로 *이동*.

> 저자의 농담 — *"dev 가 SEMANTIC_UI 에서 LEGACY_UI 로 파일을 옮기면, 해 질 녘까지 의례용 당나귀 모자를 써야 하고, Slack Shamebot 이 부여하는 수치 점수를 받아야 한다."*

### § 4.6 기존 기능 마이그레이션 — 스크립트 + 어댑터 + 수동

#### § 4.6.1 직접 해서 보여주기

> *"팀을 빠르게 끌어들이는 가장 빠른 방법은 호의를 부탁하지 않고 마이그레이션이 그냥 일어나게 만드는 것."*

확장이 안 된다고 느껴질 수 있지만, *툴링* 이 들어오면 된다.

#### § 4.6.2 로컬 린트

레거시 UI 코드를 빨리 찾는 법 — *린터를 다시 사용*. 단, 모든 IDE 가 경고를 토하지 않도록 *로컬에서만* 실행.

#### § 4.6.3 프리미티브 마이그레이션은 자동화 안전

> *"인라인 값을 프리미티브로 교체하는 것은 거의 항상 안전한 베팅."*

```
view.layer.cornerRadius = 8         →  Radius.r8
label.textColor = UIColor.darkGray  →  Palette.greyDarkest
```

기계적이라 자동화 가능. 스크립트가 끝나면 *마이그레이션된 PR* 을 다른 팀에 보낸다.

#### § 4.6.4 그러나 — 시멘틱 마이그레이션은 자동화 위험

핵심 함정. 같은 raw 컬러가 *여러 시멘틱 역할* 에 쓰일 수 있다.

예: `UIColor(hex: 0xFFFFFF)` 를 *맹목적* 으로 `Color.primaryBackgroundColor` 로 교체하면 — `background` 가 아니라 `surface` 또는 `highlight` 인 경우 *틀려서 버그*. 다크 모드에서 컬러가 잘못된다.

#### § 4.6.5 80/20 + Undo 워크플로

> *"대부분의 경우 white 가 background 라고 가정. 모든 `UIColor(hex: 0xFFFFFF)` 를 맹목적으로 `Color.primaryBackgroundColor` 로 교체."*

결과: 80% 는 맞고 20% 는 틀림. *그러나 이제 남은 20% 만 수동으로 고치면 된다 — 100% 가 아니라!*

워크플로: 스크립트 → `git status` 변경 *수동 검토* → 잘못된 20% *되돌리기(undo)*. 한 단계 더 — 그 20% 중 절반이 `highlight` 라면, 모두 `Color.highlight` 로 마이그레이션 후 나머지 절반 undo. *이제 90% 가 스크립트로*.

#### § 4.6.6 어댑터로 중간 단계

*UI 프리미티브 업데이트는 알지만, 시멘틱 값이 아직 미정* 일 때.

```swift
enum MigrationAdapter {
    static var backgroundColor: UIColor { Palette.lightBlue }
    static var selectedColor:   UIColor { Palette.lightBlue }
}
```

둘 다 같은 `Palette.lightBlue` 로 해소되지만, *발산할 유연성* 과 *임시 시멘틱 상태* 를 준다. 나중에 최종 컬러로 교체하면 된다.

어댑터가 유용한 경우: 시멘틱 이름이 아직 정의되지 않음, 같은 raw 값이 여러 역할에 쓰임, 잘못된 시멘틱 가정 없이 코드를 정리하고 싶음.

> **Note** — 어떤 팀은 UI 마이그레이션을 *피하려고* 복잡한 *swizzling 런타임 핵* 으로 다크 모드를 지원하려 했지만, 일이 엄청나게 복잡해지고 시멘틱 의미를 추론할 수도 없게 된다.

### § 4.7 마이그레이션을 *채택* 으로 — 심리학 10가지 도구

이 챕터에서 가장 무게가 실린 절. 코드 이야기 절반, 사람 이야기 절반.

> *"많은 부분이 심리학이다 — 저항이 어디서 오는지, 개발자가 무엇을 피하고 싶은지, 어떻게 마이그레이션을 그들의 워크플로의 일부로 만들지."*

#### § 4.7.1 컴포넌트 우선 — *가장 큰 레버리지*

공유 컴포넌트(`PrimaryButton`, `CardView`, `FormField`) 가 있다면 *완벽한 진입점*. 모든 개발자에게 `Color.primaryBackground` 를 기억하라고 요청하는 대신, *공유 컴포넌트가 내부적으로* 그걸 쓴다. 개발자는 평소처럼 `CardView` 를 쓰고 *생각 없이* 올바른 UI 요소를 받는다.

```swift
// Before — 9 줄 Button 구성
// After
PrimaryButton("Delete") { /* ... */ }
```

#### § 4.7.2 Status quo bias

> *"사람들은 자기가 아는 것을 선호한다 — 버그가 있어도."*

레거시 UI 가 완벽하지 않아도 *익숙하고 편안* 하다. *나란히 비교(side-by-side comparison)* 로 옛 코드와 새 코드를 *직접 보여준다*.

#### § 4.7.3 새 규칙 — *Touch a view? Migrate it.*

> *"전면 마이그레이션을 위해 개발을 일시 중지하는 것은 비현실적."*

증분 접근. 누군가 UI 컴포넌트를 *건드릴 때마다* 시멘틱 UI 로 업데이트. `LoginView` 를 업데이트하면 *린터가 경고* 를 띄우고, 몇 분만 더 쓰면 그 파일을 즉시 마이그레이션할 수 있다.

> 제안 — 머지 전 *체크리스트 항목* 으로 UI 마이그레이션. 테스트 작성과 같은 식으로.

#### § 4.7.4 Loss aversion — *뒤집기*

> *"사람들은 이미 작동하는 것을 포기하는 것을 좋아하지 않는다."*

관점에서 *위험은 가치가 없다*. 그러니 질문을 뒤집는다 — *"마이그레이션하지 않으면 무엇을 잃는가?"*

- 다크 모드·더 큰 텍스트 지원이 훨씬 어려워질 것
- 새 팀원 온보딩이 느리고 지저분하게 유지
- 업데이트를 *공짜로* 받지 못함
- *나머지를 막을 것* — 아이콘 업데이트 자동 적용 불가, 새 아이콘 롤아웃 차단

> *"트릭: 그대로 머무는 비용이 변경 비용보다 무겁게 느껴지게 만들기."*

#### § 4.7.5 작은 한 걸음

> *"시작이 가장 어려운 부분. 한 작은 단계를 밟으면 계속할 가능성이 훨씬 크다."*

첫 단계를 작게. *"`LoginButton` 을 새 `PrimaryButton` 으로 교체. 그게 다야."* 페어 프로그래밍도 좋다. 그리고 *축하* — *"Profile 화면이 100% 시멘틱 컬러를 씁니다!"*

#### § 4.7.6 Social proof

> *"아무도 첫 번째 사용자가 되고 싶지 않지만, 마지막도 되고 싶지 않다."*

다른 팀이 이미 새 시스템을 쓰면 갑자기 *안전하게* 느껴진다. 과시한다.

- *"Onboarding 팀이 마이그레이션 후 컬러 업데이트를 공짜로 받습니다!"*
- *"Settings 팀이 시멘틱 텍스트 스타일을 사용 — 접근성을 개선했습니다!"*
- *"전체 채택까지 두 팀 남았습니다. 가자!"*

#### § 4.7.7 Self-Determination Theory — 자율성·역량·연결성

심리학 이론에서 빌려온 세 욕구.

| 욕구 | 적용 |
|---|---|
| **자율성** (I choose to do this) | 한꺼번에 강제하지 말고 점진적 *opt in*. 도구를 제공하고 규칙은 적게. |
| **역량** (I know how to do this) | 명확한 예시, 변경 전후 비교, 업그레이드 가이드. 소설 분량의 문서 없이도 성공 가능하게. |
| **연결성** (I feel connected) | *전환한 다른 팀의 이야기* 공유. *"혼자 하지 않는다"* 는 신호. |

#### § 4.7.8 IKEA effect

> *"사람들은 자기가 만든 것에 더 신경 쓴다 — 일부만 만들어도."*

얼리 어답터가 새 UI 시스템에 *기여* 하도록 — 컴포넌트 명명, 간격 이슈 수정, 새 변형 제안. 그들은 그것을 *자기 것* 으로 느낄 것이다.

#### § 4.7.9 보상 & 인정

먼저 가는 팀을 *축하* — 사내 뉴스레터, Slack 외침, 팀 미팅 강조. *데이터로 뒷받침* — 시간 절약, 코드 깨끗함. *인정은 마이그레이션을 추가 작업이 아니라 진보로 느끼게 한다*.

#### § 4.7.10 — 결국 사람들이 새 시스템을 *원하게* 만들기

> *"마이그레이션은 기술 도전이 아니라 매우 심리적인 도전이다."*
>
> *"사람들은 게으르거나 저항하지 않는다 — 조심스럽고 바쁘다. 현재 시스템에서 전달하는 법을 배웠고, 전환은 그 안전망을 포기하는 것을 의미한다."*
>
> *"목표는 위에서 아래로의 마이그레이션 계획으로 사람들을 밀어붙이는 것이 아니라 — 새 시스템을 안전하고 명백한 다음 단계로 느끼게 하는 것."*

---

### § 4.8 앱 전반 마이그레이션 — 컴포넌트 어댑터·API 일치·Feature Flag

뷰별 마이그레이션이 *혼합 UI(전환 상태)* 를 만들 때, 한꺼번에 적용하는 기법들.

#### § 4.8.1 컴포넌트 어댑터

`OldPrimaryButton` 을 `PrimaryButton` 으로 마이그레이션할 때 — *400 개 콜 사이트* 를 교체하는 대신, `OldPrimaryButton` 을 그대로 두고 *API 동일*. 내부 구현만 swap.

```swift
struct OldPrimaryButton: View {
    var body: some View {
        PrimaryButton(action: action) { Text(label) }
    }
}
```

수백 버튼을 한 번에 마이그레이션, 화면 코드 변경 없이. *`OldPrimaryButton` 이 마이그레이션 어댑터* 가 된다.

#### § 4.8.2 API 를 일치시키기

`OldPrimaryButton(label:, action:)` ↔ `PrimaryButton(label:, action:)`. *이름 변경만으로* 옛 버튼 제거 가능. 미래의 마이그레이션을 위한 작은 친절.

#### § 4.8.3 Feature flags — *폰트는 다른 이야기*

어댑터는 레이아웃·컬러·공유 컴포넌트엔 좋다. 폰트는 다르다. 기본 본문 폰트를 바꾸면 *거의 모든 화면* 에 잔물결.

```swift
if FeatureFlag.useNewFont {
    return self.font(.custom("Proxima-Nova-Bold", size: 32, ...))
}
return self.font(.system(size: 22, weight: .regular))
```

충분한 뷰가 `companyFont(style:)` 을 사용한 뒤 *플래그를 켜면* 새 헤더가 한 번에. *플래그 삭제 전* 에는 신·구 두 변형이 모두 안정함을 확인.

---

### § 4.9 결론 — *플레이북은 UI 너머에도 적용된다*

> *"다룬 단계 대부분이 시각 너머에도 적용된다. 다른 라이브러리로 마이그레이션에도 같은 접근이 통한다."*

새 분석 설정, 크래시 로거, 새 네트워킹 레이어로 전환 — 같은 도전에 부딪힌다.

- 다른 사람의 마찰 줄이기
- 자동화 기회 찾기
- 수동 처리 결정
- 스크립트 작성
- 새 코드가 *올바른 도구* 를 쓰도록 보장 (린팅 룰)
- 앱 전반 어댑터 사용

*플레이북은 놀랍게도 일관된다*.

## 나누고 싶은 이야기

### 1. 인상 깊은 부분 — *다른 팀원들을 설득하는 챕터*

이번 두 챕터에서 가장 강하게 박힌 건 *Ch.4 가 사실은 코드 챕터가 아니라 심리 챕터* 였다는 점이다. 기술서를 읽는다고 생각하고 펼쳤는데, 절반은 *사람을 어떻게 움직일 것인가* 에 대한 책이었다.

특히 § 4.7 의 흐름 — *컴포넌트 우선 → status quo bias → Touch a view migrate it → loss aversion 뒤집기 → 작은 한 걸음 → social proof → SDT → IKEA effect → 보상*. 익숙한 개념들도 있지만, 이 묶음을 *마이그레이션이라는 한 가지 과제* 에 일관되게 엮어놓으니 갑자기 그림이 또렷해졌다.

평소엔 *"왜 다른 팀이 우리 라이브러리를 안 쓸까?"* 같은 질문을 *기술적 결함* (API 가 안 맞는다, 문서가 부족하다, 성능이 떨어진다) 의 관점으로만 봐왔다. 그런데 저자는 잠시 입장을 바꿔보라고 한다. *지금 당신은 기능 엔지니어다. 마감에 쫓기고 있다.* 그 순간 모든 기술적 우월성은 의미가 없다. *"지금 필요한 게 그게 아니다"* 라는 한 문장이 모든 걸 무력화시킨다.

#### 가장 마음에 남은 문장

> *"사람들은 게으르거나 저항하지 않는다 — 조심스럽고 바쁘다. 현재 시스템에서 전달하는 법을 배웠고, 전환은 그 안전망을 포기하는 것을 의미한다."*

이게 § 4.7 전체를 한 줄로 요약한다. *게으름이 아니라 안전망 포기에 대한 두려움*. 이 시선만 잡으면 나머지 도구들이 자연스럽게 따라온다.

- *왜 린터 메시지 문구가 중요한가* — 문구가 처벌처럼 느껴지면 안전망을 포기할 동기가 더 사라진다.
- *왜 first-mover 가 어려운가* — 안전망을 가장 먼저 포기해야 하니까. 그래서 *social proof* 가 결정적.
- *왜 IKEA effect 인가* — 새 시스템이 *내 안전망* 의 일부가 돼야 포기 비용이 사라진다.
- *왜 Touch a view → migrate it 인가* — *어차피 안전망을 한 번 풀어야 하는 그 순간* 에 마이그레이션을 끼워 넣는 것.

#### 실무로 가져오면

마이그레이션 발표를 할 때 *"이거 더 좋은 거예요. 다 같이 옮깁시다"* 만으로는 부족하다. 저자가 알려준 사고 순서로 다시 짜야 한다.

1. *내 발표를 듣는 사람은 누구인가* — 기능 엔지니어다. 마감에 쫓기고 있다.
2. *그들이 잃는 게 무엇인가* — 익숙한 코드. 작동하는 코드. 안전망.
3. *어떻게 비용을 낮추는가* — 직접 마이그레이션 PR 을 보낸다 (§ 4.6.1). 어댑터로 콜 사이트를 보존한다 (§ 4.8.1). *작은 첫 걸음* 을 만든다.
4. *어떻게 머무는 비용을 키우는가* — *마이그레이션 안 하면 다크 모드 못 받고, 새 아이콘 못 받고, 온보딩 비용이 늘어난다* (§ 4.7.4 loss aversion 뒤집기).
5. *어떻게 외롭지 않게 하는가* — 먼저 간 팀을 *큰 소리로 축하한다* (§ 4.7.6 social proof + § 4.7.9 보상).
6. *어떻게 *자기 것* 으로 느끼게 하는가* — 얼리 어답터에게 명명 권한, 변형 제안 권한을 준다 (§ 4.7.8 IKEA effect).

평소 *"기능 엔지니어가 협조해 주지 않는다"* 라고 답답해했던 상황들이, 사실은 *내가 그들의 입장에서 비용·이익을 다시 그려보지 않았던* 문제였다는 걸 인정하게 된다. 이 깨달음이 이번 정리에서 제일 크다.

> 한 줄로 요약하면 — *마이그레이션은 코드를 옮기는 일이 아니라, 사람들이 안전하다고 느끼는 지점을 옮기는 일* 이다.

### 2. Ch.3 — *모든 UI 요소가 같은 패턴은 아니다*

Ch.2 에서 폰트·컬러를 *프리미티브 + 시멘틱* 두 층으로 잡으면서 *"이 패턴이 모든 UI 요소에 통한다"* 고 기대했다. Ch.3 가 그 기대를 살짝 깨준다.

- *Spacing* 은 2 층이 아니라 *3 층*. 동적 spacing 까지 잡아야 폰·태블릿 차이를 흡수한다.
- *Icons* 는 *시멘틱과 프리미티브를 둘 다 노출* 해도 OK. 컬러와 정반대 권고.
- *Shadows* 는 반대로 *프리미티브를 노출하지 않음*. 시멘틱만으로 충분.

처음엔 *"왜 이렇게 일관되지 않지?"* 싶었는데, § 3.4 의 두 질문 휴리스틱(*직접 입력? + 사례 예측?*) 을 보고 정리됐다. *패턴은 일관된다 — 답이 다를 뿐*. 각 요소의 특성에 따라 두 질문의 답이 갈리고, 그 답이 노출 패턴을 결정한다.

이게 *멘탈 모델의 본질* 이라는 점에서 위로가 된다. 새 UI 요소(border, corner radius, motion duration, z-index 등) 가 추가돼도 두 질문만 던지면 어디에 위치시킬지 바로 답이 나온다. *암기할 룰이 아니라 적용할 사고 도구* 를 받은 셈.

### 3. Compose 매핑 — 어댑터·feature flag 도 그대로 통한다

책 예제는 SwiftUI 기반이지만, 이번 두 챕터의 핵심 기법들은 *플랫폼 중립* 이다.

#### Spacing

```kotlin
object Space {
    val s4 = 4.dp
    val s8 = 8.dp
    // ...
}

object Spacing {
    val xs = Space.s8
    val sm = Space.s16
    val md = Space.s24
    val lg = Space.s32
}

Column(verticalArrangement = Arrangement.spacedBy(Spacing.md)) { /* ... */ }
```

동적 spacing 은 `WindowSizeClass` (Material 3 의 window-size adaptive API) 와 결합한다. SwiftUI 의 `@Environment(\.horizontalSizeClass)` 와 거의 1:1 대응.

#### Icons — Compose 도 *시멘틱 + 프리미티브 공존*

```kotlin
object Icons {
    val close          = R.drawable.ic_cross
    val dismiss        = R.drawable.ic_cross
    val remove         = R.drawable.ic_cross
    val navigationBack = R.drawable.ic_arrow_back

    // 프리미티브 직접 노출도 OK
    val cross     = R.drawable.ic_cross
    val arrowBack = R.drawable.ic_arrow_back
}
```

`R.drawable` 직접 사용도 살아남는다. 보편적 표준(`Icons.Default.ArrowBack` 등) 이나 일회성 자산은 카탈로그를 더럽히지 말고 그대로 두면 된다. *비율로 사고하기* — 시멘틱 70~80%, Material 표준 10~20%, R.drawable 직접 5~10%.

#### 컴포넌트 어댑터

콜 사이트 보존이라는 아이디어가 그대로 적용된다.

```kotlin
@Composable
fun OldPrimaryButton(
    label: String,
    onClick: () -> Unit,
) {
    PrimaryButton(label = label, onClick = onClick) // 내부만 swap
}
```

수백 곳의 호출부 코드를 그대로 두고, *Old* 의 내부 구현만 새 컴포넌트로 위임. 이름이 같으면 — 또는 `typealias OldPrimaryButton = PrimaryButton` 같은 트릭이면 — 최종 제거도 한 번에.

#### Feature flag

Compose 에선 더 깔끔하게 떨어진다. `CompositionLocal` 또는 ViewModel-flag 로 묶어 `MaterialTheme` 의 `typography` 슬롯에 어느 토큰 세트를 주입할지 결정. 한 번 켜면 *전체 트리* 가 갈아탄다.

```kotlin
val typography = if (FeatureFlag.useNewFont) NewTypography else LegacyTypography
MaterialTheme(typography = typography) { /* app */ }
```

같은 *Listing 4.20* 의 아이디어가 Compose 의 디자인과 더 자연스럽게 맞는다.

### 4. *린팅의 심리* — 처벌 vs 가이드

Ch.4 의 작은 디테일 하나가 자꾸 떠오른다.

> *"경고 메시지가 단지 잘못을 지적하는 것이 아니라, 도움이 되거나 관련 문서로 링크하도록 만드세요. 문구의 중요성을 과소평가하지 마세요. 이는 메시지가 처벌이 아닌 도움 가이드처럼 느껴지게 합니다."*

평소 린트 룰을 짤 때 *"이 라인은 금지된 패턴입니다"* 정도로 끝냈다. 그런데 같은 룰이라도 *"`Color.kt` 의 `primaryBackground` 사용을 권장합니다. 마이그레이션 가이드: <링크>"* 처럼 *대안과 맥락* 을 같이 주면, 받는 사람이 *왜 이게 떠있는지* 를 즉시 안다. 그리고 더 중요한 건 — *이 룰을 만든 사람이 나를 도우려 한다는 신호* 가 코드에 박힌다.

이게 § 4.7 의 SDT *역량(competence)* 과 정확히 맞물린다. 린트 경고가 *"잘못했어"* 가 아니라 *"이렇게 하면 돼"* 로 읽히는 순간, 받는 사람의 *역량감* 이 깎이지 않는다. 도구 자체가 *심리적 안전망* 의 일부가 된다.

작은 디테일 같지만, 이런 미시적 결정들이 누적돼서 *마이그레이션이 처벌처럼 느껴지는 팀* 과 *진보처럼 느껴지는 팀* 의 차이를 만든다는 게 책의 일관된 주장이다.

### 5. 정리 — *Ch.3 은 멘탈 모델, Ch.4 는 사람*

두 챕터를 합쳐서 보면 흐름이 또렷하다.

- **Ch.3** 은 UI 라이브러리의 기초를 *마무리* 한다. 폰트·컬러에 spacing·icons·shadows 를 더해 *5 가지 UI 프리미티브* 가 완성된다. *프리미티브 vs 시멘틱* 멘탈 모델이 일반화된다.
- **Ch.4** 는 그 라이브러리를 *모두에게* 보낸다. 그런데 이 *모두* 라는 단어 안에 *수십, 수백 명의 다른 우선순위와 두려움과 안전망* 이 들어 있다는 점을 책이 끝까지 잊지 말라고 한다.

> *"코드베이스 마이그레이션은 사람들이 살고 있는 동안 집을 리노베이션하는 것과 같습니다."*

이 한 문장이 Ch.4 전체를 요약한다. *집을 더 좋게 만드는 건 기술적 과제다. 하지만 사람들이 그 안에 계속 살고 있다는 사실이 그 일을 완전히 다른 종류의 과제로 만든다*.

다음 Ch.5 에서는 이 단단해진 UI 라이브러리를 *토큰* 으로 끌어올려 진짜 디자인 시스템으로 가는 단계가 기다리고 있다. *기초 → 마이그레이션 → 토큰화* 의 흐름이 한 줄로 보인다.
